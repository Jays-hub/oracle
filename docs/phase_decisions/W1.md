## Key Design Decisions

### Decision: a flat, denormalized CSV format for the self-serve BOM upload
- **Chose:** one CSV with columns `dish_name, ingredient_name, qty, recipe_unit, canonical_unit,
  yield_factor` — the same shape as the seam's `BomRow`, minus ids.
- **Over:** reusing the CLI's normalized 3-file model (`sample_dishes.csv` /
  `sample_ingredients.csv` / `sample_recipe_lines.csv`), which joins on hand-authored UUIDs.
- **Because:** a chef doing a self-serve upload has dish names and ingredient names, not UUIDs to
  invent and keep consistent across three files. `dish_id`/`ingredient_id` are derived from the
  names via `normalize_name()` instead — the same join key already used elsewhere in this codebase
  to match sales to dishes. This is a genuine simplification of the capture act for a real chef,
  not just an implementation shortcut.
- **Risk if this was wrong:** if a future integration needs the CLI's 3-file shape as the web
  upload's *input* format too (e.g. a POS system that already exports UUID-keyed data), this
  format would need a second upload path. Believed low-risk: the CLI format is a fixture/demo
  format from Phase 0, not a real POS export shape either.

### Decision: stateless upload→confirm round-trip via base64-encoded hidden form fields
- **Chose:** `POST /upload` parses + validates, then renders a confirm page carrying the *original
  uploaded bytes*, base64-encoded, in two hidden `<input>` fields. `POST /confirm` decodes,
  **re-parses and re-validates from scratch**, then writes.
- **Over:** server-side staging (write the uploaded files to a temp directory keyed by a token,
  read them back on confirm).
- **Because:** rule 07 requires stateless handlers ("shared state lives in the store, not in
  process memory"). A ~15-25 item CSV is a few KB — trivially small to round-trip through a form
  field — so there is no server-side staging state to manage, clean up, or leak across requests.
  `/confirm` never trusts the round-tripped payload blindly; it re-validates as if it were a fresh
  upload (tested — `test_confirm_rejects_tampered_payload_never_trusts_the_hidden_field`).
- **Risk if this was wrong:** if a future upload format grows past a few hundred KB (e.g. a
  multi-thousand-line invoice history), round-tripping raw bytes through hidden fields stops being
  thin and this would need real staging. Not a concern at the stated ~15-25 item scale.

### Decision: accumulate every row error, don't fail fast
- **Chose:** `parse_sales_csv`/`parse_bom_csv` collect **every** row's validation error and return
  them all together.
- **Over:** the CLI's existing pattern in `_export_to_raw` (pre-refactor), which raised on the
  first bad row and aborted the whole export.
- **Because:** a chef fixing a CSV by hand needs to see everything wrong in one pass, not
  fix-resubmit-fix-resubmit one row at a time. This is strictly better UX for the exact same
  validation rules, at no extra schema cost.
- **Risk if this was wrong:** none identified — this is additive; fail-fast behavior is a subset of
  what accumulate-all reports.

### Decision: extract `write_seam_atomic()` into `src/capture/seam_upload.py` and have the CLI
  (`src/run.py`) call it too, instead of building a second atomic writer just for the web path
- **Chose:** one shared, atomic (temp-file + `os.replace`) Parquet writer, used by both the CLI's
  `_export_to_raw` and the new web confirm handler.
- **Over:** leaving the CLI's direct `to_parquet()` calls alone and only making the *new* web path
  atomic.
- **Because:** rule 05 ("reuse the single definition... a parallel copy would silently drift") and
  rule 07 ("writes are atomic and idempotent where possible") both applied to the write itself, not
  just to the new caller. Since I was already building the atomicity logic for W1, duplicating it a
  second time inside `run.py` would have created exactly the kind of drift risk this codebase has
  already been burned by once (the `food_cost_pct` reconciliation regression, `W0_review.md`
  MINOR-2). The CLI keeps its own row-construction logic (different input shape: 4 normalized
  files vs. one flat upload); only the persistence step is shared.
- **Risk if this was wrong:** low — `_export_to_raw`'s only test (`test_export_paths.py`) checks
  path resolution, not the write mechanism, so the refactor is behavior-preserving and verified by
  the full suite (`make check`, 249/249).
- **Bug caught by this refactor's own tests:** copying `src/store.py`'s
  `parents[3] = repo-root` comment into `seam_upload.py` without adjusting for the extra `capture/`
  directory level initially resolved `RAW_DIR` one level too shallow (into `onramp/`, not the repo
  root) — caught immediately by a manual smoke test before any test was even written, then locked
  in by `test_raw_dir_invariant`. This is the same class of off-by-one the repo has hit once before
  (`test_export_paths.py`'s docstring describes the historical `_REPO_ROOT` bug in `run.py`).

### Decision: the success page explicitly does NOT claim `/` now shows the uploaded data
- **Chose:** `success.html` states plainly that the grid at `/` still shows shared sample data, and
  that reading a tenant's own uploaded data back into a dashboard is a later phase's job.
- **Over:** a generic "success!" message with a link back to `/`, implying the just-uploaded data
  is now visible there.
- **Because:** it isn't. `/` (`web/compute.py`) reads `onramp/plate_cost/data/sample_*.csv`, not
  `data/raw/`, by W0's own deliberate design (the seam has no prices, so it can't reconstruct
  margins). Confirming a W1 upload writes real seam files but changes nothing about what `/` shows.
  Rule 06 ("Honest Data Presentation... a confidently-wrong figure loses the chef on day one")
  reads to me as being exactly as much about honest *claims* as honest *numbers* — implying a
  connection between two screens that doesn't exist would be the UI-trust equivalent of a
  false-precision bug.
- **Risk if this was wrong:** none — this is the more conservative, more honest choice; the
  alternative (an unqualified success link) is strictly more misleading.

## Constraints Encountered Mid-Build

- **`python-multipart` was entirely absent** from the environment and both requirements files.
  FastAPI needs it for any `UploadFile`/`Form` parsing; nothing before W1 exercised that path.
  Installed into `restaurant-dev` and added to `requirements.txt` (with the new W1 section) and
  `requirements.lock.txt` (single line, hand-inserted — see the aside below).
- **No file-size/MIME-type upload conventions existed anywhere in the repo** to reuse. Rule 07 says
  uploads must be size-limited and type-checked but names no numbers. Invented
  `MAX_UPLOAD_BYTES = 1_000_000` (1 MB) — generous for a ~15-25 item sitdown, documented inline in
  `seam_upload.py`, easy to revisit.
- **Regenerating `requirements.lock.txt` via a full `pip freeze` pulled in unrelated conda-installed
  packages as local `file://` paths** (numpy, scipy, lightgbm, scikit-learn, and transitive deps —
  none of which this phase touched). Reverted the full regen and hand-inserted the single new
  `python-multipart==0.0.32` line instead, to avoid polluting the lock file with machine-local paths
  that would break on a different install.
- **`_validation_message()`'s first draft crashed on cross-field validator errors.**
  `SalesExportRow`'s `period_end < period_start` check is a `@model_validator(mode="after")`, whose
  pydantic error has an **empty** `loc` tuple (no single field to blame) — `err["loc"][0]` raised
  `IndexError` the first time a test exercised that path
  (`test_parse_sales_csv_accumulates_every_row_error_not_just_the_first`). Fixed by falling back to
  the bare message when `loc` is empty. Left in as a comment at the fix site.

## Reviewer Focus Areas

1. **The base64-hidden-field round-trip as the confirm mechanism.** This is the least conventional
   piece of this build — no staging store, no session, no cookie. I believe it's correct and
   rule-07-compliant (stateless, always re-validates, tested for tamper-resistance), but it's a
   pattern this codebase hasn't used before and is worth a skeptical look, particularly: is there
   any request-size limit at the FastAPI/Starlette layer that could reject a legitimately-sized
   confirm POST before my own `MAX_UPLOAD_BYTES` check ever runs on the *original* upload? (I
   didn't find one, but didn't exhaustively verify Starlette's defaults.)
2. **The invented `MAX_UPLOAD_BYTES = 1_000_000` constant and the BOM/sales CSV column contract.**
   Both are new policy I made up because nothing prior existed to reuse (confirmed via the
   Explore-agent scan). They're documented and easy to change, but a reviewer should sanity-check
   whether 1 MB and the six-column BOM format are the right calls versus, say, a config-driven
   limit or a format that also captures `menu_price` (deliberately excluded here to match the
   seam's existing no-prices contract, per `data/CONTRACT.md`).

## Post-Review Remediation (2026-07-02)

`/review-web W1` returned **"Yes, with one MAJOR fix"** (full report: `docs/phase_decisions/W1_review.md`).
Jay greenlit all findings. All three actionable findings fixed; the fourth (known-deferred) and the
NIT were left as the reviewer explicitly said no fix was required for W1.

- **[MAJOR fixed] Sample-data banner over the operator's own data.** `base.html`'s banner is now
  `{% block sample_banner %}...{% endblock %}` (defaults to the existing text, so `grid.html` needed
  zero changes). `upload.html`, `confirm.html`, `success.html` override it empty — this confirmed
  Reviewer Focus #1 was actually a real gap, not just a stylistic worry: the review reproduced the
  banner rendering directly above a real dish name on the confirm page. New regression test
  `test_capture_funnel_pages_never_show_the_sample_data_banner` asserts the banner is absent on all
  three capture pages and still present on the grid.
- **[MINOR fixed] `/confirm` had no size policy of its own.** This directly resolved my own
  Reviewer Focus #1 question about a Starlette-layer size limit — the review found it: Starlette's
  `FormParser` (used because `confirm.html`'s form has no `enctype`, so it POSTs
  `application/x-www-form-urlencoded`) caps any single field at 1,048,576 bytes by default, and
  FastAPI calls `request.form()` with no override hook, so that cap isn't reachable to raise from
  application code without bypassing FastAPI's `Form()` convenience entirely. Rather than fight the
  framework default, shrunk `MAX_UPLOAD_BYTES` from 1,000,000 to 700,000 — `700_000 * 4/3 ≈ 933 KB`
  stays safely under the 1,048,576-byte cap with margin, so **nothing that passes `/upload`'s own
  check can ever be rejected by Starlette at `/confirm`** — and added the same `_size_errors()` check
  (extracted as a shared helper) directly in `confirm_submit`, so `/confirm` enforces its own policy
  even when POSTed to directly, never relying on `/upload` having already run. New tests: a policy
  invariant (`test_max_upload_bytes_stays_under_the_base64_inflated_form_field_cap`) that fails loudly
  if this constant is ever raised without re-checking the math, plus a route test that POSTs directly
  to `/confirm` (never touching `/upload`) with an oversized payload and confirms the friendly 422.
  Verified by hand against the real server: a 650 KB file (under the new limit) now completes
  upload→confirm→save without dead-ending, where it previously would not have at anything above
  ~768 KB.
- **[MINOR fixed] Seam write was atomic per file, not jointly across the bom/sales pair.** Split
  `_atomic_write_parquet` into `_stage_parquet` (serialize to a temp file, no destination touched)
  and a commit step in `write_seam_atomic` that stages *both* files before renaming *either* into
  place. This closes the exact failure the review reproduced (fail the second `to_parquet` call,
  end up with a new bom.parquet paired with a stale/missing sales_export.parquet) because now a
  failure during serialization of either file happens before any destination file is touched — the
  only remaining inconsistency window is the gap between the two `os.replace` calls, which are two
  near-instant syscalls rather than a full DataFrame serialization. Strengthened the existing
  `test_write_seam_atomic_leaves_no_partial_file_on_failure` to also assert `bom.parquet` doesn't
  exist after a failure (it would have, pre-fix — the original test only checked the sales side and
  so didn't catch the mismatched-pair bug the review found). Added
  `test_write_seam_atomic_leaves_old_pair_untouched_on_failure`, reproducing the review's exact
  repro: seed an old pair, fail a new write partway through, assert the OLD pair survives byte-for-byte.
- **[not fixed, per reviewer] Mutable-name-derived seam ids** — reviewer explicitly said "None
  required for W1; track under the CONTRACT's stable-item-id note." Left as-is; already recorded in
  this file's Explicitly Deferred table.
- **[not fixed, per reviewer] Import-time `sys.path` mutation / row-number citation on multi-line
  CSV fields** — reviewer explicitly marked both optional/cosmetic ("Leave as-is"). Left as-is: the
  `sys.path` pattern mirrors the existing, deliberate `run.py` convention, and the row-number
  behavior is correct for the realistic case (a chef editing the CSV in a spreadsheet app, where
  each record is one row regardless of embedded newlines) — only mis-cites for someone counting
  physical lines in a raw text editor.
- **Suite after remediation: 253 tests, 253 pass** (`make check`: lint clean, both import-linter
  contracts kept). Up from 249 — net +4 (two atomicity tests, one size-policy invariant test, one
  banner regression test).
