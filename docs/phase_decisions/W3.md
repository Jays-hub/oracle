## Key Design Decisions

### Decision: Digital-feed CSV over photo+OCR for invoice capture
- **Chose:** a structured CSV upload (`src/capture/invoice_upload.py`) mirroring the W1 capture
  funnel's parse/cross-reference/write shape.
- **Over:** building `src/ingestion/`'s photo + OCR + learned-mapping pipeline.
- **Because:** OCR requires new external dependencies/services never vetted or chosen; the roadmap
  itself calls entity resolution "the hardest part"; and the doc explicitly names a digital vendor
  feed as an equally valid Phase-2 entry point. This is the thinnest slice that actually delivers
  the invoice/price-history leg end-to-end.
- **Risk if this was wrong:** under-delivers if a real chef's invoices only arrive as photos, not
  exportable line items. `src/ingestion/`'s own gate note is untouched, so that heavier build still
  has a clearly marked home when it's taken on.

### Decision: `price_observations.csv` accumulates (append + idempotent dedupe), not full-replace
- **Chose:** `write_price_observations_atomic` reads any existing file, concatenates new rows,
  de-duplicates on `(ingredient_id, observed_date, unit_price, source_invoice)`, then does the same
  temp-then-rename atomic commit `seam_upload.py` uses for a single file.
- **Over:** `seam_upload.py`'s full-replace "current snapshot" model (right for BOM/sales, which are
  a point-in-time recipe sheet and covers window).
- **Because:** Phase 3's entire point is week-over-week price trend detection, which requires
  retained history, not a snapshot — confirmed by both the CONTRACT and the ingestion stub's own
  docstring.
- **Risk if this was wrong:** the file grows without bound and has no retention/versioning policy —
  recorded as an explicit deferral below, not silently ignored.

### Decision: entity resolution is a name-key match + non-blocking warning, not a learned-mapping queue
- **Chose:** `ingredient_id = normalize_name(ingredient_name)` (the same convention `BomRow` already
  uses), plus `cross_reference_ingredients()` — a non-blocking warning listing invoice ingredient
  names that match no known BOM ingredient, surfaced on the confirm page and mirroring
  `cross_reference_dishes`'s existing honesty pattern exactly.
- **Over:** the roadmap's "confirmation queue with learned mappings" (a persistent, improving
  match table).
- **Because:** that system is explicitly the flagged-hardest part of the *gated* OCR path; building
  it now for the CSV path — before the simpler mechanism has even been used once — is exactly the
  sophistication-before-the-simpler-step pattern the Anti-Drift order names.
- **Risk if this was wrong:** a chef with an inconsistently-spelled ingredient gets no persistent
  memory of a prior correction — every invoice re-flags the same near-miss. Judged acceptable for a
  single-operator tool at this stage; noted for the reviewer.

### Decision: `/insights` reports an ingredient-cost delta only, never a margin/food-cost-tier claim
- **Chose:** `Opportunity`/`AffectedDish` carry only `prior_cost`/`new_cost`/`delta` on ingredient
  cost; the headline text is literally the roadmap's own example shape ("Beef +16% this week — N
  dishes affected") minus the margin-tier clause.
- **Over:** the roadmap's fuller example ("...short rib is now your thinnest-margin entrée").
- **Because:** `BomRow` (the on-disk seam shape) has no `menu_price` field — verified directly in
  `schemas/seam.py` — the identical constraint that already blocked a costed `/` grid (W0) and a
  per-dish `/your-data` breakdown (W2) from showing real margins. Reporting a tier/margin claim
  without a menu price would be a fabricated number (rule 06's false-precision guard), not an
  honest one.
- **Risk if this was wrong:** the surface reads as less punchy than the roadmap's literal example.
  A future phase that captures `menu_price` (the still-deferred "Co provenance" derived leg) can
  extend `Opportunity` with a real margin/tier claim once that data actually exists.

### Decision: "ceteris paribus" costing when isolating one ingredient's price move
- **Chose:** `build_opportunities` holds every *other* ingredient a scoped dish needs at its single
  latest known price, varying only the moving ingredient between the "prior" and "current" cost
  calculation.
- **Over:** trying to reconstruct each dish's total cost at two different real calendar dates
  (which would require every ingredient in that dish to have a synchronized price history — not
  realistic for how invoices actually arrive).
- **Because:** the finding this surface exists to make is "how much did THIS ingredient's move
  affect these dishes," not "how has this dish's total cost drifted from every source" — the latter
  is a different (also useful, not built) report.
- **Risk if this was wrong:** if two ingredients in the same dish move significantly in the same
  window, that dish appears once per moving ingredient, each holding the other's move constant
  rather than compounding them. A deliberate simplification, flagged below for reviewer awareness,
  not a silent bug.

### Decision: reuse `seam_upload.py`'s private parse/stage helpers rather than forking a copy
- **Chose:** `invoice_upload.py` imports `ParseResult`, `_decode`, `_validation_message`, and
  `_stage_parquet` directly from `.seam_upload` (underscore-prefixed but same-package siblings).
- **Over:** copy-pasting an equivalent implementation into the new module.
- **Because:** rule 05's reuse discipline, and because these four pieces of logic (accumulate-all-
  row-errors parsing scaffolding, UTF-8-BOM-tolerant decode, pydantic error formatting, and the
  stage-then-rename atomic write primitive) are exactly the same shape needed here — duplicating
  them would be a drift risk (two copies to keep in sync) for zero benefit.
- **Risk if this was wrong:** a future refactor of `seam_upload.py`'s "private" helpers could break
  `invoice_upload.py` silently since Python doesn't enforce the underscore convention. Acceptable
  same-package coupling; flagged so the reviewer knows it's deliberate, not an oversight.

## Constraints Encountered Mid-Build

- `src/ingestion/__init__.py` already existed as a **docstring-only stub** carrying its own
  self-referential gate note. Left it completely untouched; the new capture path lives in
  `src/capture/invoice_upload.py` instead — a deliberate architectural fork, not an accidental
  duplicate module.
- `schemas/seam.py` / `schemas/__init__.py` / `schemas/README.md` all previously and explicitly
  marked `price_observations.csv` as "still deferred (Anti-Drift)". All three needed updating now
  that the leg actually crosses — a three-file, easy-to-miss update surface for any future new leg.
- **Discovered mid-build:** `src/store.py`'s `_RAW_DIR` and `web/app.py`'s own `_RAW_DIR` are two
  *separate* module-level constants. A web test that writes seam data directly (bypassing the app's
  own write routes) and then reads it back through a route that calls `store.read_*()` must
  monkeypatch **both** independently, or the read silently falls through to whatever the real
  `data/raw/` happens to contain. This wasn't exercised by any existing W2 test (no prior test wrote
  + read-back through the web layer's `store` calls in the same test) — two of this phase's own new
  insights tests hit exactly this gap before being fixed. Flagged below for the reviewer as a
  pre-existing test-isolation blind spot, not something this phase introduced.
