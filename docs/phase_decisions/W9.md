## Why this, why now — and why it was built speculatively

W9's own spec names an explicit trigger: **"a second real tenant's data needing to coexist."** No real
second tenant exists yet — the platform is still pre-discovery (`CLAUDE.md`: "Simulation and the
on-ramp's later phases remain pending real customer discovery"). Per `.claude/rules/00-process.md`,
building is never gated, but I flagged this explicitly to Jay before starting, since the phase's own
ordering rationale in `website_production_overview.md` says partitioning a store that holds one
restaurant is "speculative" — building it now means guessing at a shape real tenant data would
otherwise dictate. Jay's directive (this session): **build it anyway, as forward infrastructure,
picking a reasonable scheme and flagging clearly that the shape is a guess.** That is what follows.

Why *this* is the natural next step regardless: W0–W8 are built and reviewed; `data/CONTRACT.md`'s own
W2 forward note already named this exact gap (`data/raw/` flat and single-tenant) as the one thing
standing between today's PoC and `create_account`'s single-restaurant fence lifting (`src/auth/
service.py`, `docs/phase_decisions/W5_review.md` BLOCKER-1). `Identity.restaurant_id` has flowed through
every W5+ route for two phases specifically as forward-plumbing for this moment (`src/auth/service.py`
docstring: "seam-level tenancy is W9").

## Practices invoked, in all three domains

- **(a) Software/coding craft — layering as a defect-prevention structure, not just organization.**
  The write-side compute (`seam_upload.py`, `invoice_upload.py`, `tenant_grid.py`) needed *zero*
  changes because it was already designed to take an opaque `raw_dir: Path` rather than reach for a
  global. Tenancy resolution lives in exactly one place per peer (`store.tenant_raw_dir()` on read;
  `web/app.py`'s route handlers on write, where `identity.restaurant_id` already exists). This is the
  "prevent the defect by construction" standard from the build brief applied concretely: a cross-tenant
  leak would require either passing the wrong path at the one resolution point (loud, single-purpose,
  easy to review) or a compute function reaching for a global it doesn't have access to (structurally
  impossible — none of them import `store`).
- **(b) Data-science/statistical concept — the firewall as an information-boundary discipline, not
  just an access-control one.** The same whitelist-beats-blacklist posture that keeps `data/_truth/`
  from ever reaching a model (rule 01) is what `_assert_raw_only`'s parent-name check now extends one
  level deeper: a path is refused unless it structurally *cannot* be anything but a named tenant's own
  subdirectory. This mirrors the censored-demand and leakage-canary disciplines elsewhere in the engine
  — the guard fires at the boundary, not hoped-for by convention downstream.
- **(c) Restaurant/consulting standard — tenant isolation as the operator's actual promise.** "Your
  data is yours, and no one else's touches it" is the literal claim `/your-data`'s transparency page
  already makes (W4). Before this phase, that claim was true only by accident (there was exactly one
  restaurant's data in the store); W9 makes it true by construction, which is the difference between an
  honest promise and a lucky one the moment a second kitchen signs up.

## Key Design Decisions

### Decision 1: per-tenant subdirectory (`data/raw/<restaurant_id>/`), not a `restaurant_id` column
- **Chose:** `data/raw/<restaurant_id>/{bom,sales_export,price_observations,food_cost}.parquet` (on-ramp
  legs) and `data/raw/<restaurant_id>/{pos_sales,...}.csv` (simulator legs) — one subdirectory per
  tenant, `data/raw/` itself now a container, not a file store.
- **Over:** a `restaurant_id` column added to `schemas/seam.py`'s `BomRow`/`SalesExportRow`/etc., with
  every reader filtering by it.
- **Because:** (1) every existing writer's concurrency/replace semantics stay untouched — a full
  replace of "my tenant's bom.parquet" is still exactly correct once the path has one more segment; a
  column would make every full-replace writer wrong (it would delete every OTHER tenant's rows from a
  shared file) and force a rewrite into read-modify-write with new locking, mirroring
  `price_observations.parquet`'s existing `fcntl` lock but now needed on 3 more files. (2) It makes
  cross-tenant leakage a **file-system-visible, structurally preventable** class of bug — a leak would
  require passing the wrong *path*, not forgetting a `WHERE` filter at one of the 8+ read call sites
  spread across `forecasting/src/evaluate/*.py` and the on-ramp's web layer. (3) The two producers
  (on-ramp Parquet legs, simulator CSV legs) use disjoint filenames within the same tenant slot today,
  so there is zero collision risk co-locating them.
- **Risk if this was wrong:** the alternative (column) has real advocates too — it avoids ever having
  8+ small subdirectories and keeps `forecasting/`'s loader reading one physical location regardless of
  tenant count. If a real second tenant's actual usage pattern makes cross-tenant *aggregate* queries
  common (e.g. "total covers across all restaurants"), a column is easier to query across; a
  subdirectory scheme would need an explicit fan-out loop. Not needed today — no such cross-tenant
  aggregate exists or is planned.

### Decision 2: path-safe-slug validation (`^[A-Za-z0-9_-]{1,64}$`), not strict UUID parsing
- **Chose:** `store.tenant_raw_dir()` validates `restaurant_id` against a permissive alphanumeric/
  hyphen/underscore pattern before building a path.
- **Over:** `uuid.UUID(hex=restaurant_id)` (stricter — matches what a real `Restaurant.id` always is).
- **Because:** a real `restaurant_id` reaching this function always originates from `identity.
  restaurant_id` (DB-issued, always a valid UUID hex in production), so the *security* need is
  narrower than "must look like a UUID" — it's "must not be a path-traversal string" (`..`, `/`, `\`,
  null bytes, empty). The wider suite already has a long-standing, widely-copied test convention of a
  fixed literal `restaurant_id="r1"` (`test_web_upload.py`, `test_web_invoice.py` — not something this
  phase should force a rewrite of just to satisfy a stricter-than-necessary check). The permissive
  pattern still fully defends the actual threat (traversal) while not forcing every test fixture in the
  suite onto a UUID literal.
- **Risk if this was wrong:** if a future need arises to enforce "every restaurant_id really is a
  Restaurant.id" at the store layer (rather than trusting the caller), this would need tightening. Low
  risk today: `identity.restaurant_id` is the only production source, and it is always a real UUID hex.

### Decision 3: `data/_truth/` is NOT tenant-partitioned by this phase
- **Chose:** leave `data/_truth/` flat; `RestaurantSimulator.run()`'s `truth_dir` parameter keeps its
  existing default untouched.
- **Over:** tenant-scoping truth the same way raw was scoped, for full consistency (the generator writes
  both in one call).
- **Because:** `_truth/` is explicitly not part of the seam `data/CONTRACT.md` governs — it's scoring-
  internal to `forecasting/src/evaluate/`, walled off by a completely separate law (rule 01's "single
  funnel for the oracle"). `website_production_overview.md`'s W9 row never names it. Doubling the
  blast radius into `evaluate/`'s internals (which would need explicit truth_dir arguments threaded
  through 8 files, on top of the raw-side changes) is exactly the kind of scope-creep the Anti-Drift
  Standing Order warns against — nothing today ever simulates two tenants at once, so there is no live
  bug, only a latent inconsistency if that ever changes.
- **Risk if this was wrong:** if a second tenant is ever actually *simulated* (as opposed to captured
  live through the on-ramp) for a real dollar-floor comparison, running the generator twice would have
  the second run's truth silently overwrite the first's in the shared flat `_truth/` directory. Flagged
  in `data/CONTRACT.md` and here as a forward note, not built. **This is the phase's single least
  certain scope call — reviewer should sanity-check it explicitly.**

### Decision 4: the shared sentinel is the nil UUID (`"0" * 32`), independently hardcoded in both peers
- **Chose:** `"00000000000000000000000000000000"` (`uuid.UUID(int=0).hex`) as the fixed bucket both the
  on-ramp CLI (`src/run.py::_DEMO_RESTAURANT_ID`) and the forecasting engine
  (`loader.py`/`generator.py::SIMULATED_RESTAURANT_ID`) write/read by default, with no shared import
  (the peer boundary forbids one).
- **Over:** inventing a human-readable constant like `"demo"` or `"default"`.
- **Because:** the nil UUID is a well-known reserved sentinel in the UUID spec for exactly this "no real
  identity" case, and — since it's a valid 32-char hex string — it satisfies the *same* validator
  `tenant_raw_dir()` applies to every other restaurant_id, with no special-casing. `data/CONTRACT.md`
  records the literal so a future reader can diff both peers' copies against one source of truth.
- **Risk if this was wrong:** none identified — this is a naming choice with no behavioral risk either
  way.

## Constraints Encountered Mid-Build

- **Found and fixed a real, unrelated regression in `scripts/backup.py` (W7).** `backup_raw_dir()`
  only ever copied files directly under `RAW_DIR` (`if item.is_file()`), never subdirectories. Once
  every tenant's data moved one level deeper, this would have **silently backed up zero files** on
  every future run — a real production-hardening feature quietly going dark, with no error, no test
  failure (the existing test only checked flat files). Fixed to `shutil.copytree` each tenant
  subdirectory; added `test_backup_raw_dir_copies_tenant_subdirectories` as the regression guard. This
  wasn't named in the phase spec but is squarely downstream of the seam-shape change this phase makes —
  leaving it broken would have been a silent defect this exact build introduced.
- **The `_read_raw_parquet` error message needed a second pass.** My first draft built the "file not
  found" message from `path.relative_to(RAW_DIR.parent)` — which breaks under every test's
  `monkeypatch.setattr(store, "RAW_DIR", tmp_path)` pattern (the real, unpatched `RAW_DIR.parent` no
  longer matches). Caught by running the suite, not by inspection; fixed to build the message from
  `base_dir.name` (the tenant segment) — a fixed, human-facing "data/raw/..." hint independent of
  where `RAW_DIR` actually resolves, matching the original hardcoded-prefix convention this replaced.
- **`test_web_auth.py`'s helper restructuring was the single largest mechanical cost.** Tests using
  `create_account` (real DB-issued, random UUID restaurant_id) needed account creation reordered
  *before* seeding `data/raw/`, since the id isn't known until the account row exists — unlike the
  fixed-literal `_FAKE_IDENTITY` pattern (`"r1"`) used elsewhere, which needed no reordering. Both
  `_seed_account`/`_logged_in_client` (this file) and `test_web_dishes.py`/`test_web_menu_prices.py`'s
  equivalents were split into a `_create_account` step (returns the real id) and a separate `_login`
  step, so seeding can happen in between.
- **One edit-tool retry silently dropped a signature change.** A `web/your_data.py` edit that failed
  once on an exact-match error was retried with a narrower `old_string` that updated
  `_price_leg_stats`'s *body* but not its `def` line — caught only by running `make test`, not by
  re-reading the diff. Worth a reviewer's independent check that every function signature in the
  touched files actually matches its call sites, not just that the bodies look right.

## Reviewer Focus Areas

1. **The `data/_truth/` scope cut (Decision 3).** I read the phase spec as explicitly raw-side-only and
   left truth-scoping as a flagged forward note rather than building it. If that reading is wrong — if
   "multi-tenancy across the seam" was meant to include the simulator's truth output too — this phase
   is incomplete on its own terms, not just conservatively scoped.
2. **The permissive path-safe-slug validator vs. strict UUID parsing (Decision 2).** I chose the
   looser regex specifically to avoid forcing a rewrite of the test suite's long-standing `"r1"`-style
   fixture convention. If that reasoning is judged backwards — the test convention should have changed
   to match production's real UUID shape, not the validator loosened to match the tests — that's worth
   revisiting; it's a real (if narrow) input-validation looseness at the one place a caller-supplied
   string becomes a filesystem path.
