## Why this, why now

The service review recorded in `website_production_overview.md` (2026-07-13, `[decided]` in
`progress_log.md`) named W5 as the foundation every later phase depends on: W6 (costed views over the
tenant's own data) needs real users/restaurants to own that data; W7 (hosting) needs persistent,
revocable sessions before it's safe to expose the app beyond `127.0.0.1`; W9 (seam tenancy) needs a
real multi-restaurant concept to exist somewhere before it can be pushed into the seam. W5 is squarely
first because it is the one phase every other phase writes to or reads from.

Concretely, this phase retires three placeholders the W2 build explicitly flagged as "revisit if this
ever becomes a real multi-user credential store": the single env-configured operator credential
(`ONRAMP_AUTH_USERNAME`/`ONRAMP_AUTH_PASSWORD_HASH`, SHA-256), the `RESTAURANT_ID = "default"` constant,
and the itsdangerous-signed, unlistable, ephemeral-secret session cookie. It also picks up one item the
production overview's seam table (row 5) scoped into this same phase: staged-upload state (the
hidden-base64-form-field round trip W1/W3 built) moves server-side into the app DB.

## Practices invoked, in all three domains

- **(a) Software/coding craft.** The ORM/migration split (SQLAlchemy models as the single source of
  truth, Alembic's `--autogenerate` deriving DDL from them rather than hand-typed SQL) keeps schema and
  code from drifting silently — the same reason `test_db_engine.py` runs the *actual* migration against
  a throwaway DB rather than only diffing metadata. The layering also mirrors the existing
  `src/`-pure / `web/`-thin split: `src/auth/service.py` and `src/capture/staging.py` never import
  FastAPI, so they're unit-testable without a running server (rule 05).
- **(b) Data-science/statistical concept — none new this phase, but a security-adjacent analogue worth
  naming.** Argon2id's memory-hardness is the security-engineering cousin of the "make the attacker do
  real work per guess" idea the platform doesn't otherwise use — the KDF cost parameter is a knob (like
  a model hyperparameter) whose "right" value trades off attacker cost against the app's own login
  latency, sized to a generation-appropriate default rather than tuned here.
- **(c) Restaurant/consulting standard.** "Invite-only account creation" mirrors how a real restaurant
  group onboards a new location today — someone with authority (a CLI operator standing in for an
  account manager) vouches for the account rather than a self-serve web form, which matches the
  on-ramp's actual go-to-market motion (a person in the room, not a marketing funnel) at this stage.

## Key Design Decisions

### Decision: opaque, DB-hashed session tokens replace itsdangerous-signed cookies entirely
- **Chose:** the cookie carries only a random `secrets.token_urlsafe(32)` value; the server stores
  `SHA-256(token)` in the `sessions` table and looks it up on every request.
- **Over:** keeping Starlette's `SessionMiddleware` and just persisting `ONRAMP_SESSION_SECRET` so
  sessions survive a restart (the literal reading of production-overview row 2).
- **Because:** a signed-but-unrevocable cookie can still authenticate after "logout" if the client just
  keeps presenting it (Starlette's `SessionMiddleware` has no server-side revocation hook at all) — the
  production overview's own target state (§2.4) demands sessions the app DB can "list and revoke," which
  a signed cookie structurally cannot do no matter how the secret is managed. A DB-backed opaque token
  gets revocability, listability, *and* incidentally removes the "which process has the signing secret"
  problem altogether, since there's no signature to forge or lose.
- **Risk if this was wrong:** a DB lookup on every request is marginally slower than verifying a
  signature in-process; at today's scale (one dev/demo process) this is immaterial. If W7's hosting
  target needs sub-millisecond session checks under real load, an index on `token_hash` (already
  present) should be sufficient — no redesign needed.

### Decision: staged uploads are single-use rows in `staged_uploads`, not a cache/queue
- **Chose:** a plain SQLAlchemy table row with `consumed_at`, checked and set atomically inside the same
  request that reads it; no TTL sweep/cleanup job.
- **Over:** an in-memory cache (Redis-style) or reusing the old stateless hidden-field approach with the
  bytes simply moved into a signed token instead of a raw hidden field.
- **Because:** the production overview names this specifically as an app-DB table (§3 schema sketch),
  and a plain row is the thinnest thing that gives every property the old mechanism needed (survives a
  page reload, re-validated at confirm time) plus the one it didn't (server-side control over
  size/lifetime, no client-visible payload). No cleanup job is a deliberate, explicit gap — see
  Explicitly Deferred.
- **Risk if this was wrong:** low. Worst case, unconsumed rows accumulate slowly (bounded by how often
  a chef starts but abandons an upload) — a cosmetic app-DB growth issue, not a correctness or security
  one, and trivially fixed later with a scheduled delete-where-expired job.

### Decision: `MAX_UPLOAD_BYTES` is left at 700 KB even though its original rationale (surviving a
  1.33x base64 client round-trip under Starlette's 1 MiB form cap) no longer applies
- **Chose:** keep the number unchanged; only update the comment explaining why.
- **Over:** raising it now that the client never round-trips the payload.
- **Because:** W5 is scoped to identity + storage, not upload ergonomics, and 700 KB was already a
  deliberate, generous ceiling on its own terms (a 15-25 item recipe sitdown is a few KB). Changing a
  hostile-input boundary as a side effect of an unrelated phase is exactly the kind of scope creep the
  build-phase discipline warns against.
- **Risk if this was wrong:** none — this is a pure "did not change" choice; raising the limit later is
  a one-line, easily-reviewed change whenever an actual need shows up.

### Decision: `Identity.restaurant_id` exists on every session but is not yet used to filter any
  `data/raw/` read
- **Chose:** carry `restaurant_id` on `Session` and `Identity` (so the plumbing exists), but every
  `store.read_bom()`/`read_sales()`/etc. call remains exactly as unscoped as it was in W2-W4.
- **Over:** wiring `restaurant_id` into `src/store.py`'s reads now, "since the field is right there."
- **Because:** `website_production_overview.md` §4 explicitly assigns seam partitioning to W9, gated on
  a second real tenant, and calls it "a gated, cross-peer `data/CONTRACT.md` change" — not something an
  on-ramp-only phase should do opportunistically just because a column happens to be available. Doing it
  now would also be untestable in any meaningful way (there is exactly one tenant's data in the physical
  store), which is precisely the W2 "no physical multi-tenant partition" reasoning, still valid.
- **Risk if this was wrong:** none new — this preserves W2-W4's existing (documented, reviewed) scope
  boundary rather than expanding it.

### Decision: `is_authenticated()` (nav-only) swallows DB errors; `require_login`/`current_identity`
  (the real gate) do not
- **Chose:** the cosmetic nav flag fails toward "show the logged-out nav" on any `SQLAlchemyError`; the
  actual route guards let such an error propagate up to whatever try/except (or lack thereof) the route
  already has.
- **Over:** making every route defensively wrap its `require_login`/`current_identity` call in its own
  try/except (a bigger diff, mostly touching routes that had no failure-handling need before this phase
  introduced a DB dependency into the nav path).
- **Because:** `is_authenticated()`'s own existing docstring already called it "cosmetic only, never a
  content gate" before W5 — extending that same tolerance to its new failure mode (a broken DB) is
  consistent, not a new risk. The real gate intentionally does *not* get the same treatment: swallowing
  a DB error there would mean a broken app DB quietly presents as "you're not logged in" everywhere,
  which is confusing but not a security regression (it still denies access) — and building a distinct
  "system unavailable" surface for every route is judged out of scope for this phase (see Explicitly
  Deferred).
- **Risk if this was wrong:** if the reviewer judges the "DB outage looks like a login prompt" behavior
  unacceptable even temporarily, the fix is bounded (wrap two or three currently-unwrapped routes) —
  flagged below as a Reviewer Focus Area rather than treated as settled.

> **Post-review correction (2026-07-14, `/review-web W5` MINOR-4):** the paragraph above
> misdescribes the code that actually shipped. `web/auth.py::_identity()` — the single helper
> behind `is_authenticated`, `require_login`, *and* `current_identity` — catches `SQLAlchemyError`
> for **all three**, not just the cosmetic nav flag. A DB outage on the real gate therefore
> fail-closes to a `303` redirect to `/login`, not an uncaught 500. This is the *safer* of the two
> described behaviors, not the riskier one — but it means Reviewer Focus Area #1 below was answered

> implies was already possible. See `docs/phase_decisions/W5_review.md` MINOR-4 for the reviewer's
> full note.

## Constraints Encountered Mid-Build

- **SQLite naive/aware datetime mismatch.** SQLAlchemy's plain `DateTime` column type round-trips a
  timezone-aware Python `datetime` through SQLite as a **naive** value on read (the bind processor
  formats only the wall-clock digits, dropping any offset silently — no error). `_utcnow()` in
  `src/db/models.py`, `src/auth/service.py`, and `src/capture/staging.py` originally returned an aware
  `datetime.now(timezone.utc)`; comparing that against a value that had passed through a `db.refresh()`
  or a fresh query (now naive) raised `TypeError: can't compare offset-naive and offset-aware
  datetimes`. Fixed by making every `_utcnow()` in this phase return `datetime.now(timezone.utc)
  .replace(tzinfo=None)` — naive, but still correctly UTC — so every stored/compared timestamp uses the
  same convention. Two tests (`test_resolve_session_rejects_expired_session`,
  `test_password_reset_rejects_expired_token`) that directly mutated a live ORM object's timestamp
  needed the identical fix, since an in-session object's attribute doesn't get silently naive-ified
  until an actual DB round trip touches it.
- **Alembic's `fileConfig()` default disables every other logger in the process.** `migrations/env.py`
  calls `logging.config.fileConfig(alembic.ini)` to wire up Alembic's own log formatting; that
  function's default `disable_existing_loggers=True` sets `.disabled = True` on every Python logger
  *not* explicitly named in the ini's `[loggers]` section — including `web.app`'s. Harmless for the
  standalone `alembic` CLI (a fresh process each invocation), but `test_db_engine.py` calls
  `alembic.command.upgrade()` **in-process**, so running that one test permanently silenced `web.app`'s
  logger for the rest of that pytest session — surfaced as an order-dependent failure in
  `test_full_password_reset_flow_via_http` (which asserts on a `caplog`-captured `_log.info(...)` call)
  that passed in isolation but failed only when the full suite ran. Fixed with
  `fileConfig(..., disable_existing_loggers=False)`.
- **`itsdangerous` had no other users once `SessionMiddleware` was removed** — confirmed via repo-wide
  grep before deleting its `requirements.txt`/`requirements.lock.txt` lines, mirroring the W2 hardening
  note about not leaving stale, unused pins around (Anti-Drift: "install what the current step needs").

## Reviewer Focus Areas

1. **DB-outage behavior on the real auth gate.** ~~`require_login`/`current_identity` let a
   `SQLAlchemyError` propagate...~~ **Post-review correction (2026-07-14, `/review-web W5` MINOR-4):
   this focus area was moot before it was written.** All three — `is_authenticated`, `require_login`,
   and `current_identity` — share `web/auth.py::_identity()`, which catches `SQLAlchemyError`
   uniformly. A DB outage on the real gate fail-closes to a `303` redirect to `/login`, the same as
   the nav path, not FastAPI's generic unhandled-error response. Rule 06's "never a stack trace" bar
   holds without a further fix; the one true residual is that `POST /login` on a dead DB still
   500s (not covered by `_identity()`, since login itself calls `authenticate`, not `resolve_session`)
   — but non-debug FastAPI leaks no stack trace to the client either way, so no rule is violated.
   See `docs/phase_decisions/W5_review.md` MINOR-4.
2. **"Invite-only = a CLI script, no invite-token table" reading.** This is the most interpretation-
   dependent call in the phase — the production overview names "invite-only account creation" without
   fully specifying the mechanism. I read it as satisfied by "no public signup route," not as requiring
   an actual token-based invite flow (that felt like W10 "team access" territory, since today every
   account is independent — there's no inviter/invitee relationship to model yet). If the reviewer reads
   the spec differently, this is the design decision to push back on first.
3. **Staged-upload row lifetime with no sweep job.** Not a correctness bug (single-use consumption is
   tested and works), but worth confirming the reviewer agrees "no cleanup job yet" is an acceptable gap
   for this phase rather than something that should have shipped alongside the table itself.
