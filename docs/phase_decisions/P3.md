## Why This, Why Now (Step 0)

Phase 2's cleaned demand series (`clean_demand()`) tags a day-item `censored=True` when the kitchen
86'd it, but leaves the fed-to-the-model `demand` value at the **capped, observed** count — a lower
bound on true demand, never the real thing. Every downstream consumer (`point_floor.py`,
`day_ahead_eval.py`, `cleaning_check.py`, `baseline_floor.py`) already `.drop(columns=["censored"])`
and moves on, silently accepting that lower bound as ground truth. `point.py`'s own docstring names this
directly: *"large bias signals missing era features or censored-demand contamination — exactly the 130
censored rows currently fed into the target."* Phase 4 (the newsvendor turn) cannot be trusted on top of
a target that's wrong in a specific, systematic direction on exactly the highest-`Cu` days — so this had
to come before Phase 4, per the roadmap's own ordering.

**Codebase impact.** New files only, nothing existing modified:
- `forecasting/src/models/unconstrain.py` — `unconstrain_demand(demand_df, min_history=8)`, the
  correction itself.
- `forecasting/src/evaluate/unconstrain_check.py` — the second sanctioned oracle reader (the checkpoint:
  does recovered demand track `true_demand` on the days it was censored?).
- `forecasting/src/evaluate/unconstrain_floor.py` — the dollar gate (does an unconstrained-target point
  model beat a clean-target one on the items that actually sold out?).
- `forecasting/tests/test_unconstrain.py` — 12 tests.
- `requirements.txt` — one comment updated (scipy is now a *direct* dependency, not just lightgbm's
  transitive one).

This unlocks: a corrected target any future phase can opt into (`unconstrain_demand(clean_demand())` is
a drop-in replacement for `clean_demand()` at every existing `.drop(columns=["censored"])` call site,
though none of those call sites were rewired — see "Explicitly Deferred").

**Practices, named in all three domains:**
- **(a) Software/coding craft.** Walk-forward-safe target correction: every statistic (mean, variance,
  count) is computed via `.expanding().shift(1)` per group, mirroring `FeaturePipeline`'s own
  shift-then-roll lag discipline, so a row's correction is a pure function of its own past — provably
  safe to slice into any later walk-forward fold (`test_correction_is_prefix_stable` is the concrete
  proof, not just an assertion).
- **(b) Data-science/statistical concept.** Right-censored regression / demand unconstraining
  (revenue-management practice): fitting a count distribution (Negative Binomial, method-of-moments,
  falling back to Poisson when the sample isn't overdispersed) to an item's own uncensored history, then
  taking the **conditional expectation above the censoring point**, `E[D | D > cap]` — not the
  unconditional mean. This is the one idea the whole module hinges on; see "Key Design Decisions" below
  for why the naive-mean version this project tried first corrected almost nothing.
- **(c) Restaurant/consulting standard.** The 86-board reality: a kitchen's 86 log is a nightly-wiped
  artifact, not a permanent record. `config/sim.yaml`'s `goforward_days: 90` encodes exactly this —
  `data/raw/eightysix_log.csv` only exists for the last 90 days of a ~2.5-year history, so almost all
  historical censoring is silently invisible to the engine. This isn't a synthetic-data quirk to shrug
  off; it's the real reason Phase 1's capture had to come before Phase 3's correction, and it caps how
  much *any* unconstraining method can recover from `data/raw/` alone.

**Anti-Drift check.** The build used a real statistical technique (NegBin/Poisson tail expectation, not
a hand-wavy heuristic) because the phase spec explicitly names "Tobit/censored regression" as its
practice — this is the level of sophistication the phase calls for, not a reach past it. What was
**not** built: a full parametric Tobit MLE (Gaussian-latent-variable regression with per-row censoring
points) — the simpler count-distribution tail-expectation approach was tried first, validated
empirically against real generated data (MAE 5.076 → 3.090), and kept because it already worked; there was
no evidence a heavier method would do meaningfully better, and Anti-Drift says stop at "clearly working,"
not "maximally sophisticated."

## Key Design Decisions

### Decision: tail-conditional expectation, not the unconditional mean
- **Chose:** fit NegBin/Poisson via method of moments on prior uncensored history, then use
  `E[D | D > cap]`.
- **Over:** a plain expanding mean of prior uncensored demand (the first version built).
- **Because:** a censored day is *tautologically* a high-demand day for that item (that's why it sold
  out) — it's a selected sample from the distribution's right tail. Averaging it in with ordinary days
  regresses toward a typical day, not toward what an above-average day plausibly was. Verified this was
  a real, not theoretical, problem: on this project's actual generated data, the naive-mean version's
  average correction across all 66 real censoring events was **+0.00** — it essentially never fired,
  because the historical mean for most (item, weekday) groups sits at or below the already-near-cap
  observed value. The tail-expectation version corrected the same 66 rows by **+3.38 on average** and
  cut MAE against the hidden truth_stockouts log from **5.076 → 3.090** (bias −5.076 → −1.699).
- **Risk if this was wrong:** method-of-moments NegBin fitting is sensitive to small samples (variance
  estimates are noisy below ~10-20 observations); `min_history=8` is a judgment call, not a principled
  threshold — see "Explicitly Deferred."

### Decision: `max(observed, estimate)` — recovery can only raise the target
- **Chose:** hard floor at the observed value.
- **Over:** using the distributional estimate unconditionally.
- **Because:** a censored observation is a lower bound on true demand by construction (`observed_demand
  <= true_demand` always, per `generator.py`'s own invariant, `test_censoring_observed_le_cap`). Any
  estimate below the observed cap would be provably wrong, not just imprecise. `test_recovered_never_below_observed`
  checks this holds across a randomized series, not just the hand-picked correctness-test rows.

### Decision: expanding-window (walk-forward), not a single global fit
- **Chose:** every statistic used to build an estimate is computed from strictly-prior rows only
  (`.expanding().shift(1)` per group).
- **Over:** a single mean/variance fit over the whole series (simpler code, one fit per group instead of
  a running one).
- **Because:** a global fit would let an early row's correction depend on dates *after* it — target
  leakage of the same class rule 02-feature-eng forbids for features, just applied to a target-correction
  step instead. `test_correction_is_prefix_stable` proves the invariant directly: truncating the input to
  end exactly at a censored row's date produces the identical corrected value for that row as running on
  the full series.

### Decision: train AND score the P3 dollar gate against the corrected target, not the original capped one
- **Chose:** `unconstrain_floor.py` feeds `unconstrain_demand(clean_demand())` into
  `RollingOriginBacktest.run()` as **both** the training target and the scored "actual" — same pattern as
  `point_floor.py`, just with a different `demand_df`.
- **Over:** training on the corrected target but scoring dollar cost against the *original* capped
  `clean_demand()` actual (tested this variant directly — see below).
- **Because:** scoring a model that correctly learned to anticipate a spike against a still-capped actual
  is provably perverse — it charges "overage" for correctly predicting demand that was genuinely there,
  exactly the failure mode `baseline_floor.py`'s own docstring already warns about in the mirror-image
  case ("computing the floor on truth inflates the target and lets a mediocre model look like a winner").
  Empirically checked both ways on real data: the SAME unconstrained-trained model's predictions score
  **121,322** against the original capped actual vs. **126,363** against the corrected actual (all-items,
  n_estimators=100 config, not directly comparable to the gate's own numbers below — this was a targeted
  side experiment). The capped-actual scoring looks better specifically *because* it's the easier,
  already-known-to-be-wrong target — not because it's a fairer comparison.
- **Risk if this was wrong:** this was the single most consequential judgment call in the phase — see
  "Reviewer Focus Areas" and "Remediation" below. `/review-phase P3` found the *principle* here was
  right but the *implementation* wasn't: this build scored the clean arm against its own capped actual
  and the unconstrained arm against its own corrected actual — still two different rulers, just not the
  ones this section warns against. Fixed post-review; see "Remediation."

### Decision: fix `RollingOriginBacktest` fold placement for this gate only, don't touch the shared harness
- **Chose:** compute `min_train_weeks` dynamically inside `unconstrain_floor.py`
  (`_min_train_weeks_reaching_tail`) so folds land at the *end* of the series (train on ~everything,
  test on a tail sized to cover the go-forward window), instead of `RollingOriginBacktest`'s
  fixed-at-the-start default.
- **Over:** modifying `RollingOriginBacktest.splits()` itself to support a "recent window" anchor mode.
- **Because:** `backtest.py` is a shared P1 harness used by `baseline_floor.py`/`point_floor.py`/
  `day_ahead_eval.py`, all of which are already-reviewed, already-logged artifacts whose reported numbers
  (e.g., "$133,121.17") are cited in `progress_log.md` and `forecasting/CLAUDE.md`. Changing its default
  fold-placement behavior would silently change what those existing numbers mean without a review pass on
  that change. Scoping the fix to this one gate script (a local helper function, not a `backtest.py` edit)
  is a smaller, reviewable surface — the "build only this phase" discipline.
- **Constraint this fixed:** without it, `unconstrain_floor.py` reported a **0.00 "improvement"** —
  identical dollar cost to the cent between clean and unconstrained training — because
  `point_floor.py`-style default fold placement (`min_train_weeks=12, test_weeks=4, n_folds=4`) only ever
  scores folds in 2022-03-26..2022-07-15, and every observable censored row is dated 2024-04-05..2024-06-30.
  The two training targets never differed inside any scored fold. This was caught, not shipped — see
  "Constraints Encountered Mid-Build."

### Decision: NOT conditioning the distribution fit on menu era
- **Chose:** fit at (item, service_period, day_of_week) with a (item, service_period) fallback; no era
  dimension.
- **Over:** adding era as a third grouping key (or a fourth, alongside day_of_week).
- **Because:** `era_demand_factors` in `sim.yaml` move demand by roughly ±10% per item across eras — a
  second-order effect next to the day-of-week swing (`dow_factors` span 0.65–1.35, a much larger lever).
  Adding era would fragment the already-thin same-weekday history (a handful of eras × 7 weekdays leaves
  very few observations per cell for a ~2.5-year series) for a small expected accuracy gain. Not measured
  head-to-head — a judgment call, recorded here rather than silently made.
- **Risk if this was wrong:** if a future item's era transitions turn out to be large (a menu
  reformulation that meaningfully changes portion size/popularity), the correction would blend pre- and
  post-transition demand levels. Cheap to add later if `unconstrain_check.py`'s MAE stops looking good on
  an item with a big era jump.

## Constraints Encountered Mid-Build

- **`RollingOriginBacktest` fold placement.** Not visible from the spec or from reading `point.py`/
  `point_floor.py` alone — only surfaced by printing actual fold date ranges against actual censored-row
  dates. `point_floor.py`'s own "$133,121.17" number is therefore scored entirely on 2022 data, never
  touching a single censored row (clean or corrected) — worth knowing for anyone reasoning about what
  that number does and doesn't validate. Not a P2 bug (P2 never claimed to test the go-forward window),
  but a gap this phase had to work around explicitly rather than inherit silently.
- **`test_module_boundaries.py`'s substring scan is stricter than "no truth imports."** It textually
  greps for the literal substring `_truth` anywhere in `forecasting/src/{data,features,models,decision,report}/**/*.py`
  — including comments and docstrings. The first draft of `unconstrain.py`'s docstring (explaining the
  firewall it respects) tripped this by literally writing `data/_truth/` while explaining that it *never*
  touches it. Fixed by rewording to "the hidden ground-truth store" / "oracle" (the vocabulary
  `cleaner.py`'s own docstring already uses for the same reason). A real, if slightly ironic, catch.

## Remediation (2026-07-02, post-`/review-phase P3`)

Full findings: `docs/phase_decisions/P3_review.md`. All three findings closed:

- **BLOCKER-1 (false dollar-gate failure).** `unconstrain_floor.py` rewritten so both arms are scored
  against ONE fixed ruler (`_oracle_actual`: clean_demand()'s observed series with observably-censored
  rows raised to their real `true_demand` from the oracle — sanctioned, since this module lives in
  `evaluate/`). Each arm still trains on its own target (clean vs. unconstrained — that's the comparison),
  but the per-fold fit/predict/score loop now scores both sets of predictions against the same
  `test_actual` slice, structurally ruling out a repeat (`test_oracle_actual_is_identical_regardless_of_which_arm_it_scores`
  asserts the ruler function takes no per-arm parameter at all).
- **MINOR-1 (2-day tail-coverage gap).** Fixed surgically: `_splits_with_full_tail_coverage` extends ONLY
  the last fold's test window to the series' true final date, leaving folds 0..n-2 and the last fold's own
  start untouched. (First attempt trimmed days off the series START instead so `min_train_weeks` divided
  evenly — rejected after it shifted every fold's exact date boundaries and materially changed the dollar
  result along with the coverage; confirmed by re-running with the old fold placement and reproducing the
  reviewer's own verification number to within $8. A 2-day tail gap should not perturb the other three
  folds, and the surgical version doesn't.)
- **NIT-1 (stale MAE numbers).** Corrected in `unconstrain.py`'s docstring and above: 5.076 → 3.090 (not
  5.28 → 3.09), bias −5.076 → −1.699.
- **New regression tests:** `forecasting/tests/test_unconstrain_floor.py` (+6) guard both fixes directly —
  the ruler stays fixed regardless of which arm is scored, missing-truth rows raise loudly, and the tail
  extension touches only the last fold's end.

**Corrected result: the dollar gate PASSES.** Popular-item cost, scored on the common ruler: clean
$112,428.46 → unconstrained $110,928.54, a **+$1,499.91 improvement** (vs. the reviewer's own spot-check
of +$1,507.15 on the old, 2-day-short fold placement — the ~$7 difference is the now-included tail days,
not a discrepancy). The "touching a censored test day" decomposition also flips to a genuine win: clean
$81,690.58 → unconstrained $80,312.66. Both phase "done when" conditions are now legitimately met:
recovered demand tracks truth on capped days (MAE 5.076 → 3.090) **and** popular-item dollar cost improves
vs. Phase 2 (+$1,499.91). Verified via `make check` (232 → 238 tests, all pass; lint clean; both
import-linter contracts kept) and by rerunning `unconstrain_check`/`unconstrain_floor` against real
generated data.
