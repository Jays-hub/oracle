## Why This, Why Now (Step 0)

Every phase through P3 produced a better and better POINT estimate of demand — cleaned (P2), then
censoring-corrected (P3). But a point forecast cannot answer the question a prep sheet actually asks:
"how much should I make?" is a newsvendor problem, and the newsvendor-optimal quantity is a *quantile*
of the demand distribution (`F⁻¹(q*)`, `q* = Cu/(Co+Cu)`), not the mean. P3 had to come first because
fitting quantiles to still-censored sales fits them to a lie, worst on exactly the highest-`Cu` dishes.
This phase is the one the roadmap calls "the product in miniature" — it turns a forecast into a prep
sheet number, and per the roadmap it is "barely ML": an OR idea (the newsvendor critical ratio) plus a
quantile read-off.

**Codebase impact.** New files only, nothing existing modified except `requirements.txt`/
`requirements.lock.txt` (`mapie`, `scikit-learn` — commented-out placeholders promoted to real deps):

- `forecasting/src/models/quantile.py` — `QuantileGBMModel`: one LightGBM quantile regressor per
  requested level, global across items (mirrors `point.py`'s `GlobalLGBMModel` structure).
- `forecasting/src/decision/newsvendor.py` — `critical_ratio`, `required_quantile_levels`,
  `quantile_curve`, `prep_quantity` (the `F⁻¹(q*)` read-off), `expected_waste`, `expected_stockout`.
  Pure math, zero model/data dependencies — "model delivers quantile, policy picks which quantile."
- `forecasting/src/evaluate/calibration.py` — the calibration checkpoint: empirical coverage + PIT vs.
  `data/_truth/truth_demand.csv`, plus an independent MAPIE conformalized-quantile-regression (CQR)
  cross-check.
- `forecasting/src/evaluate/newsvendor_floor.py` — the dollar gate: quantile model + newsvendor
  read-off vs. the Phase-2/3 point model used as the mean, through the existing
  `RollingOriginBacktest`.
- `forecasting/tests/test_newsvendor.py` (22), `test_quantile.py` (10), `test_calibration.py` (8),
  `test_newsvendor_floor.py` (5) — 45 new tests.

This unlocks: a real, demoable prep-quantity number per item per day (not just a forecast), and the
waste/stockout decomposition that becomes the "free" byproduct pitch to a chef. Phase 5 (exogenous
signal fusion) and Phase 6 (hierarchy/pooling) both build on top of this distribution, not the P2/P3
point estimate.

**Practices, named in all three domains:**
- **(a) Software/coding craft.** Clean separation of model (delivers a quantile grid) from policy
  (picks which quantile and derives a decision from it) — `quantile.py` knows nothing about Co/Cu;
  `newsvendor.py` knows nothing about LightGBM. A model-agnostic calibration wrapper (MAPIE) sits
  beside, not inside, the production model, so the calibration verdict doesn't depend on trusting the
  same code path it's checking.
- **(b) Data-science/statistical concept.** Quantile regression via the pinball loss (LightGBM's
  `objective="quantile"`), non-crossing enforcement via post-hoc rearrangement (Chernozhukov, Fernandez-
  Val & Galichon 2010 — sorting each row's predictions across the quantile axis), conformalized quantile
  regression (Romano, Patterson & Candès 2019) for a finite-sample coverage guarantee, and the classic
  newsvendor→quantile bridge (`Q* = F⁻¹(Cu/(Co+Cu))`) with its waste/stockout expectations as CDF
  integrals.
- **(c) Restaurant/consulting standard.** The chef's per-dish policy *is* the critical ratio — a
  cheap-to-waste, expensive-to-run-out-of dish (high Cu/Co) gets prepped high; an expensive-to-waste,
  cheap-to-run-out-of dish gets prepped lean. Calibration is a credibility instrument for the chef ("90%
  sure" should mean right 90% of the time), and the waste/stockout split is the free-by-product pitch:
  the same number that sets the prep quantity also tells the chef how much of each failure mode to
  expect.

**Anti-Drift check.** The roadmap explicitly calls this phase "barely ML" and warns the sophistication
budget is an OR idea plus a quantile read-off — not a new architecture. What was built respects that:
one more LightGBM objective (`quantile`, already available in the already-installed library) trained at
a handful of extra levels, plus closed-form calculus for the waste/stockout integrals. What was
deliberately **not** built: a joint/monotone multi-quantile model (rule 03 explicitly sanctions the
simpler post-hoc-rearrangement fix instead), a full continuous parametric distribution fit (Tweedie/
NegBin over the whole support — the piecewise-linear quantile-grid approximation is simpler and the
calibration checkpoint confirms it's adequate), and MAPIE integration reusing `QuantileGBMModel`'s own
internal per-quantile estimators (a tighter coupling for no real benefit — see "Key Design Decisions").

## Key Design Decisions

### Decision: reimplement `critical_ratio` locally in `decision/newsvendor.py`, don't extend the import-linter carve-out
- **Chose:** a one-line duplicate of `evaluate/objective.py`'s `critical_ratio(co, cu) = cu/(co+cu)`.
- **Over:** adding `forecasting.src.decision.newsvendor -> forecasting.src.evaluate.objective` to
  `.importlinter`'s existing `ignore_imports` list (which already carries one entry for
  `models.baselines -> evaluate.objective`, with a comment framing `objective.py` as "the sanctioned
  dollar-metric shared by baselines' self-scoring").
- **Because:** the existing carve-out's own comment scopes itself to "baselines' self-scoring" — it
  reads as a narrow, single-purpose exception, not a general "anything may import objective.py" license.
  Extending the firewall's carve-out list is a bigger, more permanent structural change than duplicating
  a single closed-form formula that will not drift (`q* = Cu/(Co+Cu)` is stable microeconomics, not a
  value that could silently diverge between two copies). Anti-Drift favors the smaller, more local fix.
- **Risk if this was wrong:** if `dollar_loss`/`critical_ratio`'s formula ever changes, there are now two
  places to update instead of one. Low risk given the formula's stability; flagged for the reviewer to
  weigh in on directly, since this is a judgment call, not a forced move.

### Decision: `QuantileGBMModel` uses `lgb.LGBMRegressor` (sklearn API), not `lgb.train()`/`Dataset` like `point.py`
- **Chose:** the sklearn-compatible LightGBM wrapper for every quantile-level model.
- **Over:** mirroring `point.py`'s raw Booster/`Dataset` API exactly, and building a separate sklearn-API
  model just for MAPIE's needs.
- **Because:** MAPIE's `ConformalizedQuantileRegressor` explicitly whitelists `lightgbm.LGBMRegressor`
  (verified via a standalone smoke test, not documentation — MAPIE 1.4.1's docstring lists it alongside
  `sklearn.linear_model.QuantileRegressor`/`GradientBoostingRegressor`/`HistGradientBoostingRegressor`).
  Using the sklearn API for the entire quantile grid — not just calibration.py's own separate conformal
  fit — keeps one training code path in `quantile.py` instead of two parallel LightGBM call conventions
  in the same phase.
- **Risk if this was wrong:** none observed — the sklearn API accepts the same native LightGBM
  hyperparameter names as extra kwargs and category-dtype columns identically to the Booster API;
  verified empirically (categorical handling, non-crossing, leakage checks all behave the same).

### Decision: `evaluate/calibration.py`'s MAPIE conformal check fits its own independent pair of models, not `QuantileGBMModel`'s
- **Chose:** `conformal_coverage()` builds a fresh `FeaturePipeline` + a from-scratch
  `ConformalizedQuantileRegressor(prefit=False)` fit/conformalize/predict cycle.
- **Over:** exposing `QuantileGBMModel.estimators_`'s specific 0.1/0.5/0.9-level sub-models and passing
  them to MAPIE via `prefit=True` (which requires the caller to supply exactly the lower/upper/median
  triple in that order).
- **Because:** the whole point of a conformal cross-check is an INDEPENDENT verification that the
  production model's claimed calibration is real — reusing its own fitted artifact for the check would
  make a bug in the production model's specific 0.5-level fit invisible to the very check meant to catch
  it. The coupling `prefit=True` would require (matching alpha values exactly, exposing internals across
  a module boundary) buys nothing this phase needs; roadmap Practices (a) calls for "a model-agnostic
  calibration wrapper," which reads more naturally as "doesn't need to know how the model was built" than
  "wraps the model's own artifact."
- **Risk if this was wrong:** doubles the LightGBM training cost for calibration.py's own run (a second,
  independent 3-model fit) — acceptable for a checkpoint script, not the production hot path.

### Decision: `required_quantile_levels()` unions the standard grid with EVERY item's own critical ratio
- **Chose:** fit the model at `[0.05, 0.10, 0.25, 0.50, 0.75, 0.90, 0.95, 0.99]` union
  `{critical_ratio(item.co, item.cu) for item in items}` — 19 distinct levels on the real 11-item config.
- **Over:** fitting only the standard grid and interpolating every item's `F⁻¹(q*)` read-off.
- **Because:** rule 03-model-training.md is explicit: "a set of quantile forecasts covering [0.10, 0.25,
  0.50, 0.75, 0.90, q*]" — q* is a required, not optional, fitted level. This also makes `prep_quantity()`
  EXACT (not interpolated) at the one number the whole phase is graded on, the dollar-critical read-off.
- **Risk if this was wrong:** 19 LightGBM models instead of 8 costs real training time (the full 4-fold
  dollar gate took several minutes). Acceptable for this data size (11 items); would need revisiting at
  Phase 6's multi-restaurant scale — noted in "Explicitly Deferred."

### Decision: waste/stockout as trapezoidal integrals over a piecewise-linear quantile curve, with an explicit truncation at the top fitted level
- **Chose:** `expected_waste`/`expected_stockout` treat the fitted (quantile, value) points as anchors of
  a piecewise-linear CDF (with an explicit `(0, 0)` floor anchor for demand ≥ 0), and integrate exactly
  over that piecewise-linear approximation. The top fitted level (0.99 by default) is treated as the
  effective right tail for `expected_stockout` — probability mass beyond it is assumed negligible.
- **Over:** fitting a full continuous distribution (NegBin/Tweedie) and integrating its closed form.
- **Because:** the roadmap explicitly frames this as "barely ML... an OR idea plus a quantile read-off";
  a continuous distributional fit is exactly the kind of premature sophistication the Anti-Drift Standing
  Order warns against here. The piecewise-linear approximation is EXACT when the true CDF is itself
  piecewise-linear between the fitted points (verified directly: `test_expected_waste_and_stockout_
  brackets_a_uniform_distribution` hand-derives the closed-form Uniform(0,100) newsvendor integrals and
  checks this module's trapezoidal integration against them to a tight tolerance) and tightens as the grid
  is made finer elsewhere.
- **Risk if this was wrong:** the 0.99-truncation approximation is measurably worse near the truncation
  boundary itself (verified in the same test: ~1% relative error at Q=90 against a grid truncated at 99,
  vs. <0.1% error at interior points) — acceptable for items whose critical ratio sits well inside [0.05,
  0.95], which is true for all 11 real items (checked: the widest is 0.8125), but would need a wider or
  finer grid for a hypothetical item with an extreme cost ratio.

### Decision: `newsvendor_floor.py`'s dollar gate reuses `point_floor.py`'s simpler single-`demand_df` pattern, not `unconstrain_floor.py`'s oracle-anchored one
- **Chose:** both arms (point-as-mean, quantile+newsvendor) train on the SAME
  `unconstrain_demand(clean_demand())` target and are scored via one direct
  `RollingOriginBacktest.run(demand_df, models, items)` call.
- **Over:** building an oracle-anchored common scoring ruler like `unconstrain_floor.py`'s
  `_oracle_actual()`.
- **Because:** `unconstrain_floor.py` needed that machinery specifically because its two arms trained on
  TWO DIFFERENT targets (clean vs. unconstrained) and P3's own review (BLOCKER-1) found that scoring each
  arm against its own target's actual fabricates a false result. This phase's two arms train on the
  IDENTICAL target, so `RollingOriginBacktest.run()`'s own single `test_df` per fold is already one fixed
  ruler for both — building the oracle-anchoring machinery here would solve a problem this comparison
  doesn't have (Anti-Drift: build only what the phase needs).
- **Risk if this was wrong:** none observed — the real dollar-gate run reproduces `point.py`'s own
  previously-logged $133,121.17 point-model number exactly, confirming the shared-target/direct-`run()`
  pattern is wired correctly.

## Constraints Encountered Mid-Build

- **MAPIE's `ConformalizedQuantileRegressor` needs `alpha` passed explicitly at construction, not just
  `objective="quantile"`.** Not documented anywhere obvious — MAPIE internally checks whether `alpha` is
  present in the estimator's OWN `get_params()` output before it will clone-and-reconfigure it per
  quantile level; `LGBMRegressor(objective="quantile")` alone doesn't surface `alpha` there (LightGBM's
  sklearn wrapper only exposes constructor kwargs it was actually given). Caught via a standalone smoke
  test before writing any `calibration.py` code, not discovered mid-integration.
- **`.importlinter`'s existing carve-out comment reads narrower than a literal "objective.py is fair
  game."** This shaped the local-reimplementation decision above rather than being a late correction.

## Remediation (2026-07-05, post-`/review-phase P4`)

Full findings: `docs/phase_decisions/P4_review.md`. All 6 findings closed (1 MAJOR, 4 MINOR, 1 NIT):

- **MAJOR-1 (dollar gate never scored the go-forward/censored window).** `RollingOriginBacktest.run()`
  anchors every fold at the series START and expands forward; on this project's ~2.5-year series that
  clustered all 4 test folds in spring 2022 — the exact same root cause `unconstrain_floor.py` hit one
  phase earlier (P3_review.md BLOCKER-1). Fixed the identical way: two new free functions in
  `backtest.py` — `min_train_weeks_reaching_tail` + `splits_with_full_tail_coverage` — anchor the folds
  at the series END instead, WITHOUT changing `RollingOriginBacktest`'s own default `splits()` behavior
  (so `baseline_floor.py`/`point_floor.py`'s already-logged numbers don't move). `newsvendor_floor.py`
  now runs its own manual per-fold fit/predict/score loop against these end-anchored splits (mirroring
  `unconstrain_floor.py`'s own loop) instead of calling `RollingOriginBacktest.run()` directly.
- **MINOR-3 (made_to_order items got a dish-count Q*).** `decision.newsvendor.route_batch_items()`
  filters to `prep_type=batch` items (rule 04-deployment.md); `newsvendor_floor.py` now restricts both
  `_NewsvendorAdapter`'s item economics and the reported gate to the 7 batch items (both arms still
  TRAIN on the full 11-item demand signal — only the predicted/scored item set is batch-only).
- **MINOR-5 (single tail holdout, pooled-only coverage) + MINOR-2 (upper-tail under-coverage, unremarked).**
  `calibration.py`'s `compute_calibration()` now refits across the SAME 4 end-anchored rolling-origin
  folds the dollar gate uses (rule 03's ">=4 folds", not a single split), reporting pooled AND per-fold
  coverage. Added `per_item_underage_at_critical_ratio()` — rule 03's per-item check, which immediately
  surfaced real per-item spread the pooled number was hiding (see below). `main()` now prints an explicit
  directional caveat whenever upper quantile levels are under-covered, instead of leaving it to be
  noticed in a table. Reporting under the `day_ahead_eval.py` realistic-horizon replay (the review's
  "consider," not a required fix) was deliberately NOT built — see "Explicitly Deferred" below.
- **MINOR-4 (expected_stockout's truncation bias, undocumented).** Strengthened the docstring in
  `decision/newsvendor.py` with the quantified bias direction/magnitude and an explicit "do not surface
  this to an operator without documenting the bias" note — no math change, since Phase 8 (the only
  operator-facing surface for this number) doesn't exist yet and widening the grid now would be
  premature per Anti-Drift.
- **NIT (critical_ratio duplication).** Reviewer agreed with the original call — no action taken.
- **New regression tests (+37 total; suite 316 → 353):** `test_backtest.py` (+6, the new end-anchored
  helpers), `test_newsvendor.py` (+3, `route_batch_items`), `test_newsvendor_floor.py` (+1, a
  made_to_order item never gets a dish-count row through the adapter), `test_calibration.py` (net +3:
  −2 dead `_train_test_tail_split` tests removed since that helper was superseded, +5 for
  `per_item_underage_at_critical_ratio`).
