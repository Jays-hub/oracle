## Why this, why now

W0 (read-only reveal) and W1 (self-serve capture funnel) both explicitly deferred the same three
items to "W2 — auth + persistence": auth/tenant isolation on `/upload` and `/confirm`
(`docs/phase_decisions/W0.md`, `W1.md` Explicitly Deferred tables), the grid reading a tenant's own
uploaded data back through the seam, and wiring `src/store.py` (the DuckDB-over-Parquet helper built
in the 2026-06-29 hardening pass) into the web layer for the first time — until now nothing in
`web/` ever called it. W2 is the natural next step because W1 made the capture funnel self-serve but
left it wide open (anyone who finds `/upload` can write to the seam and see nothing shown back for
it) — auth is what turns "a chef *can* upload" into "a chef's own data is theirs."

## Key Design Decisions

### Decision: auth is a single env-configured operator credential, not a user table
- **Chose:** one username + a SHA-256 password hash, both read from environment variables
  (`ONRAMP_AUTH_USERNAME`, `ONRAMP_AUTH_PASSWORD_HASH`) at verification time. No signup, no
  password reset, no roles.
- **Over:** a real user table (even a one-row one) with a proper slow KDF (bcrypt/argon2), or an
  auth-as-a-service integration (OAuth/social login).
- **Because:** this is genuinely a one-operator tool today — there is exactly one restaurant's data
  in the store, and nothing has validated a need for more than one login. Rule 07 already mandates
  "secrets in env, never in code"; env-configured credentials are the thinnest thing that satisfies
  "AuthN/AuthZ on every data path" without building machinery (a users table, migrations, a
  password-reset flow) that nothing today needs (Anti-Drift: `website_vision.md` §9 explicitly
  disclaims a multi-tenant platform mandate). `verify_credentials` fails **closed** — if the env vars
  are unset, no login can ever succeed, so there's no accidental default/hardcoded credential.
- **Risk if this was wrong:** if a second real operator account is needed before a proper user
  store exists, this credential model has no path to "two users" without a rewrite. Believed
  low-risk short-term: the very next phase that needs a second tenant is exactly the phase that
  should build a real user/tenant table, not retrofit one onto this placeholder.

### Decision: physical multi-tenant partitioning of `data/raw/` is NOT built here
- **Chose:** keep `data/raw/` flat and unpartitioned; "isolation" is enforced entirely at the auth
  layer (every data-bearing route requires a valid session) rather than by scoping file paths to a
  tenant id.
- **Over:** adding a `restaurant_id` column to `schemas/seam.py`'s `BomRow`/`SalesExportRow`, or
  namespacing `data/raw/` into per-tenant subdirectories (`data/raw/<restaurant_id>/bom.parquet`).
- **Because:** (1) there is exactly one tenant's data in the physical store today, so the actual
  threat this build needs to defend against is *anonymous access*, not *cross-tenant leakage* —
  auth-gating every route already closes that; (2) a physical seam-layout change is a
  `data/CONTRACT.md`-governed decision touching **both** peers (`forecasting/src/data/loader.py` and
  `forecasting/src/simulate/generator.py` both hard-code a flat `data/raw/` today), not something an
  on-ramp-only phase should decide unilaterally; (3) `.claude/rules/05-fullstack-architecture.md`
  calls the current store "essentially single-tenant" by design and `website_vision.md` §9
  explicitly warns against building the multi-tenant platform before discovery validates the
  product. Recorded as a forward note in `data/CONTRACT.md` (mirrors the existing Co-provenance and
  stable-item-id forward-note pattern) so the deferral is deliberate and visible, not silently
  skipped.
- **Risk if this was wrong:** if real multi-restaurant data needs to coexist soon, this is the
  first thing a reviewer or Jay should sanity-check — I believe the repo's own rules and vision
  docs support this reading strongly, but it's the single biggest architectural call in this phase.
  **Flagged in Reviewer Focus Areas below.**

### Decision: `GET /` (the sample grid) stays public, unauthenticated
- **Chose:** no auth gate on `/` — it's the only route that stays exactly as W0 left it.
- **Over:** gating everything behind login now that auth exists.
- **Because:** `/` renders the shared illustrative "Marco" sample data (`web/compute.py`), never a
  tenant's real data — there is nothing to isolate there, and gating it would break W0's whole
  reason to exist: "show a client in 60s" *without* an account, in the room, before they've signed
  up for anything (`website_vision.md` §1). `/your-data`, `/upload`, and `/confirm` are exactly the
  routes that touch or reveal a tenant's real data, so those are the ones gated.
- **Risk if this was wrong:** none identified — `/` never touched the seam before and doesn't now;
  this preserves W0's existing tests and behavior exactly.

### Decision: session state via Starlette's `SessionMiddleware`, checked with a plain function
  (`require_login`), not FastAPI's `Depends()` + raised-exception pattern
- **Chose:** `if (redirect := require_login(request)): return redirect` at the top of each
  protected route, mirroring this codebase's existing style of handling the not-OK path as an
  explicit early return (see `grid()`'s try/except → `error.html`, or `/confirm`'s repeated
  `if errors: return ...`).
- **Over:** a FastAPI `Depends(require_login)` that raises `HTTPException`.
- **Because:** consistency with the rest of `web/app.py`, which never uses FastAPI's dependency
  injection for control flow — every existing "not OK" path is a manual check + early return. A new
  pattern here would be the only exception-based control flow in the file.
- **Risk if this was wrong:** none significant — purely a style choice; `Depends()` would work too.

### Decision: `/your-data` shows counts, not a per-dish breakdown
- **Chose:** dish count, recipe-line count, sales-row count, total covers, and the sales period —
  read straight off the DataFrames `src/store.py` returns.
- **Over:** a per-dish table or a costed grid built from the captured seam data.
- **Because:** the seam carries no prices (see Load-Bearing Assumptions), so a costed grid from
  real captured data is not possible yet — that's still `web/compute.py`'s sample data at `/`. A
  per-dish *uncosted* table (just names + counts) was considered but rejected as not clearly more
  useful than the aggregate counts for a transparency page whose job is "what did we collect and
  why," not "here's your menu" (that framing is `website_vision.md` §2 step 1, arguably a W3+
  concern once real dish-level display is wanted).
- **Risk if this was wrong:** low — this is additive; a future phase can extend the summary to
  per-dish detail without changing the page's structure.

### Decision: CSV export (not Parquet) for "your data"
- **Chose:** `/your-data/export/{bom,sales}` return CSV, generated via `store.read_bom()`/
  `read_sales()` → `pandas.DataFrame.to_csv()`.
- **Over:** returning the raw Parquet file directly (the actual on-disk seam format).
- **Because:** `website_vision.md` §4 names both "CSV/Parquet" as the promised open formats, but a
  chef wants to open this in Excel/Numbers, not need a Parquet reader — CSV is the more operator-
  legible choice for a one-click download link, consistent with rule 06's "speak the operator's
  language" discipline. The underlying data is unchanged; only the export encoding differs from the
  on-disk format.
- **Risk if this was wrong:** none significant — trivially extensible to also offer a `.parquet`
  variant later if a technical user asks for it.

## Constraints Encountered Mid-Build

- **`itsdangerous` was entirely absent** from the environment and both requirements files (mirrors
  the W1 `python-multipart` gap) — Starlette's `SessionMiddleware` imports it lazily and fails at
  middleware-construction time without it. Installed into `restaurant-dev`, added to
  `requirements.txt` (new W2 section) and hand-inserted into `requirements.lock.txt` in alphabetical
  order (not a full `pip freeze` regen — same reasoning as W1's note about avoiding machine-local
  `file://` path pollution).
- **FastAPI rejects a `Union[Response, Response]` return-type annotation by default.** Every route
  that can return either a rendered page/file *or* a `RedirectResponse` (e.g.
  `HTMLResponse | RedirectResponse`) failed at **app-startup** (route registration), not at request
  time, with `FastAPIError: Invalid args for response field!` — FastAPI tries to build a Pydantic
  response-model field from the annotation unless told not to. Fixed by adding
  `response_model=None` to each such route's decorator, which keeps the (accurate) Union type hint
  for a human reader while telling FastAPI not to attempt response-model generation from it. This
  surprised me enough that it's worth flagging for the reviewer as a place worth a second look —
  the ideal check for this ("do the declared route signatures work") is `make check` actually
  importing and exercising `web.app`, which it does via the test suite import, but it's the kind of
  startup-only failure a narrower unit test could miss.
- **httpx's `TestClient` silently drops empty-string form field values.** A parametrized isolation
  test that POSTed `{"sales_csv_b64": "", "bom_csv_b64": ""}` got a 422 ("Field required") instead
  of the expected 303 redirect, because httpx omits empty-string values from the
  `application/x-www-form-urlencoded` body it builds — the fields never reached the server at all.
  Fixed by using a non-empty placeholder (`"x"`) in the test; noted here since it's a non-obvious
  test-tooling gotcha that could resurface in future form-based tests.

## Reviewer Focus Areas

1. **The "no physical multi-tenant partition" call.** This is the single biggest architectural
   decision in this phase — I chose auth-only isolation over a `restaurant_id`-partitioned seam,
   reasoning that the existing rules/vision docs support it and that a seam-layout change is a
   cross-peer decision out of scope for an on-ramp-only phase. If this reasoning is wrong, it means
   W2 didn't really build "single-tenant isolation" as the spec intends, just an auth wall in front
   of the same flat store — worth a skeptical read of the "Physical multi-tenant partitioning"
   decision above and the `data/CONTRACT.md` forward note it produced.
2. **The ephemeral session secret (`secrets.token_hex(32)` per process when `ONRAMP_SESSION_SECRET`
   is unset).** This means every server restart invalidates all logged-in sessions — acceptable for
   today's single dev/demo process (there's no deployment story yet per `W0.md`'s still-open
   Reviewer Focus Area), but it's untested (no test restarts the process to verify the invalidation
   behavior) and worth confirming this tradeoff is the right one rather than requiring the env var
   to be set unconditionally.
