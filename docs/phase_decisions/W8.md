## Why this, why now

`website_production_overview.md` §1 row 11 named the gap directly: `GET /` is the *only* public
surface and it's just the sample-data demo grid — there is no storefront a prospect can find and
judge without us in the room, and rule 06's responsive + WCAG-AA mandates have never had a scheduled
pass. §4's ladder puts W8 right after W5/W6 on purpose: a design pass repaints every screen it
touches, so it has to land *after* those screens exist functionally (W4–W6 shipped under the
pre-W8 styles) and it shares W7's trigger (the first remote/pilot deploy) because hardening and
polish both become worth doing at the same moment — the day a stranger can actually reach the site.

## Practices invoked, in all three domains

- **(a) Software/coding craft:** presentation/logic separation held strictly — no new `src/`
  code, no new route, no new decision logic in Python; the one piece of "logic" this phase adds
  (the value-threshold comparison) is deliberately kept in the Jinja template as a `{% set %}`
  constant, not smuggled into `web/upload.py`, because it's UI copy, not business math the engine
  or the seam will ever consume. Progressive enhancement via a shared, DRY Jinja partial
  (`_progress_meter.html`) instead of copy-pasting the same 3-step markup into three templates.
- **(b) Data-science/statistical concept:** none new — a design pass on top of finished compute.
  Worth naming explicitly per the Anti-Drift Standing Order: this phase does zero modeling and
  zero forecasting work, on purpose, and that's correct, not a gap. (Named so a reader doesn't go
  looking for it.)
- **(c) Restaurant/consulting standard:** the "kitchen-hostile by default" design brief (glare,
  grease, gloves — rule 06) is a real front-of-house/back-of-house UX standard, not a generic web
  a11y checklist item: contrast and keyboard operability matter more on a tablet on the pass than
  on a laptop in an office. The "recipe-sitdown" progress meter mirrors a real service-industry
  practice — a prep list a cook can glance at mid-task to see how far through the mise en place
  they are, applied to the onboarding funnel instead of a shift.

## Anti-Drift check

This phase is *more* buildable and *more* immediately gratifying than the forecasting moat (a
CSS/copy pass has instant visible payoff), which is exactly the shape of drift `CLAUDE.md` warns
about. It is not drift here because it is the last item on a **recorded, approved** execution map
(`website_production_overview.md`, Jay's 2026-07-13 directive) with an explicit trigger and
ordering rationale — not a spontaneous detour. No sophistication was reached for beyond the spec:
no JS framework was introduced, no new backend logic, no new data model.

## Key Design Decisions

### Decision: embed the storefront hero directly in `grid.html`, keep `GET /` as the one public route
- **Chose:** add a `<section class="hero">` above the existing sample-grid markup in the same
  template/route.
- **Over:** a new `templates/landing.html` + a new route, with the demo grid moved to (or linked
  from) a separate `/demo` path.
- **Because:** `GET /` is documented as "public, no auth required (nothing tenant-specific is
  shown)" and is the only route every existing test (`test_web.py`) and the router itself treats
  as the public entry point; splitting it would touch `web/app.py` routing (a bigger surface than
  a presentation pass needs) for no functional gain, and the spec's own language — "the sample
  grid kept inside it as the clearly-labeled live demo" — reads naturally as one page, not two.
- **Risk if this was wrong:** low — if a future phase wants a dedicated marketing page separate
  from the demo, splitting `grid.html`'s hero back out into its own template is a small, isolated
  change; nothing else in this phase depends on the hero and grid staying on one page.

### Decision: heading hierarchy — quadrant headers stay `<h2>` even nested under the new "Live demo" `<h2>`
- **Chose:** hero `<h1>` (the pitch) → `<h2 id="live-demo-heading">Live demo — sample menu</h2>` →
  quadrant headers stay `<h2>Stars</h2>` etc. (unchanged from before this phase).
- **Over:** bumping quadrant headers to `<h3>` (and dish names to `<h4>`) to reflect their true
  nesting depth under "Live demo."
- **Because:** `/dishes` (the real-tenant grid) already uses the identical flat pattern —
  `<h1>Your dishes</h1>` → `<h2>Stars</h2>` sibling quadrant headers, an established convention in
  this codebase, not introduced here. Matching it on `grid.html` needed zero CSS changes (the
  `.quadrant-header h2` selector is untouched); bumping to `h3` would have required broadening
  four tag-qualified selectors (`.quadrant--star .quadrant-header h2` etc.) for a purely cosmetic
  nesting nuance. This is **not** a WCAG violation — there's no skipped level (h1→h2→h2 skips
  nothing; a violation would be h1→h3 with no h2 anywhere) — just a looser nesting than strict
  document-outline purism would prefer.
- **Risk if this was wrong:** low/cosmetic — an automated a11y checker (axe, Lighthouse) might
  flag the h2-under-h2 pattern as a best-practice nit, not a failure; fixing it later is a
  same-file, no-CSS-risk change if a reviewer wants it tightened. Flagged below as a focus area.

### Decision: a new `--focus` CSS token for the keyboard-focus ring, distinct from `--money`/`--puzzle`
- **Chose:** `--focus: #0b57d0` (a standard accessible blue), used only for `:focus-visible`
  outlines.
- **Over:** reusing `--puzzle` (`#1a4a7a`, already blue) to avoid adding a token.
- **Because:** `--puzzle` carries real semantic meaning in the quadrant legend (a "Puzzle" dish is
  high-margin/low-popularity) — reusing it for focus rings would make a focused element on
  `/dishes` visually indistinguishable from "this is a Puzzle-quadrant thing" at a glance. A
  dedicated token costs one line and removes the ambiguity.
- **Risk if this was wrong:** trivial — a single-value CSS variable, changeable anywhere.

### Decision: link styling — underline-on-text-color, not a new accent hue
- **Chose:** default `a { color: var(--text); text-decoration-color: var(--border); }`, nav links
  muted/no-underline-until-hover-or-focus.
- **Over:** leaving the browser's default blue/purple link colors (untouched before this phase),
  or introducing a dedicated "link blue."
- **Because:** rule 06 reserves "a restrained palette with a single accent for 'this is money'" —
  a stray browser-default blue on "Back to your dishes," "Set a price," etc. was never part of
  that palette and reads as an accident, not a design choice. Underline-on-neutral-text keeps
  every non-dollar interactive element visually calm and leaves `--money` as the only saturated
  accent on the page, which is the brief.
- **Risk if this was wrong:** low — purely visual; every link's `href` and behavior are unchanged,
  so a future color pass is a CSS-only change.

### Decision: responsive nav via `flex-wrap`, not a `<details>`/hamburger disclosure
- **Chose:** `.site-nav { flex-wrap: wrap; }` so the five nav links + logout button wrap onto a
  second line on narrow viewports.
- **Over:** a native `<details><summary>Menu</summary>...</details>` disclosure pattern (the
  standard no-JS accessible collapsible-nav trick).
- **Because:** the phase spec is explicit — "still server-rendered, no JS framework... a
  design-and-markup pass, not a stack change." A `<details>` menu is a legitimate no-JS pattern
  but it's a new *interaction* pattern this app has never used, and the nav only has five items —
  wrapping is simpler, requires no new component, and is fully keyboard/screen-reader operable
  with zero added markup semantics to get right.
- **Risk if this was wrong:** low/cosmetic — if five wrapped nav links look cramped on a real
  phone, swapping in a `<details>` menu later is an isolated `base.html` + CSS change.

### Decision: fix `success.html`'s stale "arrives in a later phase" copy
- **Chose:** rewrite the post-save copy to point at `/menu-prices` → `/dishes`, both of which W6
  actually built.
- **Over:** leaving it untouched as out-of-scope (it wasn't named in the W8 punch-list verbatim).
- **Because:** "the design pass across every existing screen" (the spec's own words) and rule 06's
  honesty mandate both cover this — the old copy told an operator, at the exact moment of highest
  trust (right after their first successful save), that their own margin grid "arrives in a later
  phase," which stopped being true the moment W6 shipped. Leaving a known-false claim on a screen
  a design pass is actively touching would be a silent regression this phase was positioned to
  catch, not create.
- **Risk if this was wrong:** none identified — verified `/menu-prices` and `/dishes` both exist
  and are reachable (they're in the nav for a logged-in user) before writing the new copy.

### Decision: "good enough to show value" threshold lives in the Jinja template, not `web/upload.py`
- **Chose:** `{% set _value_threshold = 3 %}` inline in `confirm.html`.
- **Over:** a named constant/function in `web/upload.py` (or `src/`) that the template reads.
- **Because:** it's UI copy — which sentence to show — not a computation the engine, the seam, or
  any other consumer will ever need. Rule 05's "controllers/glue stay thin, business logic in
  `src/`" is about *plate-cost math*; a threshold for which encouragement sentence to render isn't
  that. Keeping it in the template also keeps this phase's Python diff at zero, consistent with
  "not a stack change."
- **Risk if this was wrong:** low — if this number turns out to matter for something the engine or
  another view needs to agree on, promoting it to Python is a small, localized refactor.

## Constraints Encountered Mid-Build

- **The nav's `logged_in` state and a route's own auth gate are two independent checks.**
  `_nav_context` (a Jinja context processor run on every render) calls `is_authenticated(request)`
  directly; a test that only monkeypatches `require_login`/`current_identity` (the existing
  `test_web_upload.py` pattern) does not affect it. Discovered writing
  `test_nav_marks_the_current_page_authenticated`; fixed by patching `appmod.is_authenticated`
  too. This is pre-existing app behavior (correctly commented in `app.py` as "cosmetic only, never
  a content gate"), not something this phase changed — noted here because it wasn't obvious from
  the spec and cost one failed test iteration to find.
- **WCAG contrast verification had no automated tool available in this environment** (no
  axe-core/Lighthouke/browser). Every color pairing actually used in the templates was checked by
  hand against the WCAG 2.1 relative-luminance formula (`§ Load-Bearing Assumptions` table above);
  all passed comfortably (the tightest was 5.07:1 against a 4.5:1 requirement). This is a real
  gap relative to how a browser-based a11y pass would normally be verified — flagged below.
- **No real bound server for a live smoke test.** `ONRAMP_ALLOW_INSECURE_BIND=1 python -m web` was
  blocked by the session's own safety classifier (a correct call: that flag disarms
  `ensure_safe_bind`, a real production-hardening guard from W7, and nothing in this session
  authorized bypassing it). Verification instead used `TestClient(app)` end-to-end for every new
  behavior (12 new tests in `test_web_landing.py`), which exercises the identical ASGI app,
  middleware stack (CSRF, rate-limit, logging), and Jinja template rendering a real request would
  — the same mechanism essentially the entire existing 611-test suite already relies on.

## Reviewer Focus Areas

1. **WCAG contrast was hand-verified, not machine-checked.** I computed relative luminance /
   contrast ratios by hand for every color pairing actually rendered (money-on-white,
   money-on-bg, text-muted-on-white/bg, all three tier badges, sample-banner, skipped-notice,
   btn-primary's white-on-money) and all passed AA with margin (tightest: 5.07:1 vs. a 4.5:1
   requirement). No arithmetic tool or automated checker (axe/Lighthouse) was available in this
   environment to cross-check the calculation. If one becomes available, this is the first place
   to point it.
2. **The value-threshold number (3 dishes) is an unvalidated product judgment call.** It's cheap
   to change (one line in `confirm.html`) but nothing backs "3" specifically as the right
   chef-legible floor for "enough to show value" — it was chosen for being small, round, and
   comfortably below the vision's "~15–25 items" full-menu figure, not derived from any data.
