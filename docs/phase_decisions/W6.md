## Why this, why now / codebase impact / three-domain practices

**Why this, why now.** W5 gave the on-ramp real accounts and a designated app DB but left every
tenant staring at counts only — `/your-data` can say "12 recipe lines captured," never "your
Caesar Salad makes $6.10 a plate." The proof-of-concept's oldest, most-repeated deferral
(`data/CONTRACT.md`'s "Co provenance" note, open since before W3) is exactly this: no dish's *menu
price* has ever crossed the seam, so neither the operator nor the engine can see a real cost. W6 is
the approved next rung on `website_production_overview.md`'s ladder specifically because it needs no
new infrastructure (unlike W7/hosting) and it is "pure value — the reason the site is a *living
surface* an operator returns to." Per the ordering rationale in that doc, W6 comes before hosting.

**Codebase impact.**
- `onramp/plate_cost/src/db/models.py` — new `Dish` ORM table (the menu-price catalog).
- `onramp/plate_cost/migrations/versions/0002_add_dishes_table.py` — the matching Alembic migration
  (autogenerated, hand-verified against `compare_metadata`, renamed to the repo's `000N` convention).
- `schemas/seam.py` (+ `schemas/__init__.py`) — new `FoodCostRow`, the derived seam leg that closes
  the Co-provenance forward note.
- `onramp/plate_cost/src/costing/` (new package) — `menu_prices.py` (DB-aware catalog
  upsert/read) and `tenant_grid.py` (pure compute: the real-tenant grid, dish-detail line items,
  `FoodCostRow` derivation, and its atomic write).
- `onramp/plate_cost/src/report/grid.py` — widened `DishResult.dish_id`'s type hint
  (`UUID` → `UUID | str`) since the tenant path passes a seam string id, not a CLI UUID.
- `onramp/plate_cost/web/menu_prices.py`, `web/dishes.py` (new) — thin glue; `web/app.py` — four
  new routes (`GET`/`POST /menu-prices`, `GET /dishes`, `GET /dishes/{dish_id}`).
- `onramp/plate_cost/web/templates/{menu_prices,dishes,dish_detail,dish_not_found}.html` (new) +
  `base.html` nav links.
- `data/CONTRACT.md` — legs table + Co-provenance forward note updated to reflect the leg is now
  built (engine-side consumption is explicitly still open — out of this phase's peer).
- Downstream unlock: a later `forecasting/` phase can read `data/raw/food_cost.parquet` instead of
  `config/items.yaml`'s hand-typed `Co` placeholder — not built here (cross-peer, out of scope).

**Practices invoked, all three domains:**
- **(a) Software/coding craft.** Layered reuse over duplication: `build_food_cost_rows`/
  `build_tenant_grid` both call the *existing* `dish_ingredient_cost` (W3's degrade-non-fatal
  ingredient-cost summer) rather than forking a third implementation of the same formula;
  `write_food_cost_atomic` reuses `seam_upload._stage_parquet`'s temp-then-rename primitive (same
  cross-module private-helper reuse pattern `invoice_upload.py` already established). A duck-typed
  `_PricedDish` dataclass keeps `report/grid.py::build_grid` generic across two different id
  universes (CLI UUIDs vs. seam string ids) without forcing a shared pydantic model across them.
- **(b) Data-science/statistical concept.** None new this phase — no model, no distribution, no
  backtest. This is squarely the "barely ML" data-access-and-plumbing half of the platform charter's
  Anti-Drift note: capturing an input (menu price) and doing arithmetic (qty × price / yield) is not
  a modeling problem, and treating it like one would itself be drift.
- **(c) Restaurant/consulting standard.** Food-cost percentage and the strong/ok/thin tiering
  (`report/grid.py::food_cost_tier`) are the actual language a chef uses to judge a dish, not a
  data-science abstraction — W6 is the first phase where that vocabulary is grounded in the
  *operator's own numbers* instead of the seeded sample menu. The dish-detail "ingredient → qty →
  as-used cost" breakdown mirrors how a kitchen actually audits a plate cost by hand.

**Anti-Drift check.** This phase does not reach for sophistication before the simpler step: no new
model, no new stats. If anything the risk ran the other way — scope creep into UI/DB plumbing depth
(see Deferred, below, for where I deliberately cut it back).

## Key Design Decisions

### Decision: recompute + write `food_cost.parquet` on `POST /menu-prices`, not on every `GET /dishes`
- **Chose:** the food-cost seam leg is recomputed and written (full-replace) as a side effect of
  the menu-price save action.
- **Over:** (a) writing it on every `GET /dishes` render, or (b) tying it to `/invoice/confirm`
  (where ingredient prices actually change).
- **Because:** GET requests should not mutate state (REST convention; nothing in this codebase's
  existing routes writes to the seam on a GET). `data/CONTRACT.md`'s own Co-provenance language —
  "one recipe-confirmation act feeds two products" — gave a textual hook for treating one operator
  action as the trigger for two effects; tying it to the *menu-price save* action (rather than the
  invoice-confirm action, which predates menu-price capture and would have silently expanded W3's
  already-reviewed scope) keeps the new write path entirely inside this phase's own new code.
- **Risk if this was wrong:** the `food_cost` leg can go briefly stale relative to the latest
  invoice price if an operator uploads a new invoice without immediately revisiting
  `/menu-prices` — acceptable for a derived, best-effort engine input at this stage (no consumer
  reads it yet), but worth revisiting if/when a `forecasting/` phase starts depending on freshness.

### Decision: per-line ingredient costs on the dish-detail page are rounded to the $0.25 grid, summed (not separately rounded) into the shown total
- **Chose:** each `as_used_cost` line is `round_to_quarter(...)`; `cost_display` is the sum of
  those already-rounded lines.
- **Over:** showing cent-precision per-line costs (a genuine "audit trail" argument — rule 06's
  "provenance is reachable" language) with only the aggregate rounded separately.
- **Because:** rule 06 is explicit and quotable — "never '$6.83'" — as a blanket UI law, and I
  judged consistency with that rule (plus with `onramp/plate_cost/CLAUDE.md`'s "directional truth,
  never penny-accuracy") more important than line-level audit granularity. Critically, summing the
  *already-rounded* lines (rather than rounding the sum) guarantees the detail page's total always
  equals the sum of its own displayed lines — reconciles by eye with zero rounding-drift risk,
  mirroring the exact discipline `report/grid.py::print_grid` already documents for its own
  precise-internally/rounded-on-print split.
- **Risk if this was wrong:** a chef auditing a recipe with several cheap ($0.10–$0.30) ingredients
  may see multiple lines collapse to the same $0.25 or $0.00 figure, losing some real
  differentiation between ingredients. If this reads as a real usability problem in review, the fix
  is cheap (switch line display to cents, keep the aggregate rounded) — I did not build it that way
  by default because rule 06 reads as the stronger, more explicit constraint.

### Decision: `Dish` keyed by `(restaurant_id, dish_name)`, not a seam `dish_id` foreign key
- **Chose:** store the operator-typed display name; derive the seam join key
  (`normalize_name(dish_name)`) at read time in `menu_prices_by_seam_key`.
- **Over:** persisting `normalize_name(dish_name)` as a stored column, or a real FK to a seam
  entity.
- **Because:** the seam has no real `Dish` entity to foreign-key against (it's a denormalized CSV
  row shape, not a normalized table) — and storing a second, redundant normalized copy risks it
  drifting from the one canonical `normalize_name()` function if that function's definition ever
  changes. Deriving it fresh at read time means there is exactly one place the normalization rule
  lives.
- **Risk if this was wrong:** negligible performance cost (one Python-side dict comprehension over
  a small catalog) versus a theoretical index benefit from a stored column — acceptable at this
  scale.

### Decision: `_PricedDish` (a tiny local dataclass), not `src/bom/models.py::Dish`, feeds `build_grid`
- **Chose:** a 2-field duck-typed stand-in (`name`, `menu_price`) defined in `tenant_grid.py`.
- **Over:** reusing the CLI's pydantic `Dish` model, which already has both fields.
- **Because:** the CLI `Dish` is UUID-keyed and carries fields (`id`, `service_period`,
  `is_active`) meaningless on the tenant path — constructing fake UUIDs and default values just to
  satisfy an unrelated schema would be more confusing than a purpose-built 2-field shape, and
  would blur two genuinely different id universes the codebase already keeps separate (sample-data
  CLI vs. real seam data).
- **Risk if this was wrong:** none identified — `build_grid` only ever reads `.name`/`.menu_price`
  off whatever it's handed, so this is a safe, narrow duck-typing use.

## Constraints Encountered Mid-Build

- **The local dev app DB (`onramp/plate_cost/instance/onramp.db`, gitignored) was in a broken
  state** — it had an `alembic_version` table with no row in it, so `alembic revision
  --autogenerate` refused to run ("Target database is not up to date"). This file is scratch/dev
  state, not shared or tested against (the test suite always uses its own isolated `tmp_path`
  SQLite per `conftest.py`'s autouse fixture) — deleted it and ran `alembic upgrade head` fresh
  before autogenerating the new migration. No user data was at risk; noted here only because a
  reviewer re-running migrations locally might hit the same "not up to date" error on a stale copy.
- **`DishResult.dish_id`'s type hint (`UUID`) doesn't match reality on the tenant path.** Python
  doesn't enforce dataclass field types at runtime, so passing a `str` where the class says `UUID`
  works, but it was dishonest. Widened to `UUID | str` (see Decisions above) rather than leaving
  the annotation wrong or forking a second dataclass.
- **First draft of a test helper used the legacy `db.query(Restaurant).one()` API** instead of the
  `select()`/`db.scalar()` idiom the rest of the codebase uses everywhere else. Caught and fixed
  before handoff (`test_web_dishes.py::_set_menu_price`) rather than left inconsistent.

## Reviewer Focus Areas

1. **The `POST /menu-prices` → `food_cost.parquet` write trigger (see Decision above).** This is a
   genuine judgment call the docs left open. If the reviewer disagrees with tying it to the
   menu-price save action rather than, say, invoice confirmation or an explicit "recompute" action,
   that's the single highest-leverage design choice to push back on — it's cheap to move.
2. **The per-line rounding-to-$0.25 choice on `/dishes/{dish_id}`.** I chose consistency with rule
   06's explicit "never $6.83" over per-line audit precision. This is the one place in the phase
   where two real rule-06 concerns ("round costs" vs. "provenance is reachable") pointed in
   slightly different directions, and I picked one deliberately rather than by default — worth a
   second look.
