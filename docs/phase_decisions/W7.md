## Why this, why now

The service review (`website_production_overview.md`, 2026-07-13) named a table of eleven proof-of-
concept placeholders, each with a scheduled phase. W7 picks up rows 2 (session secrets), 3 (CSRF),
4 (hosting/TLS), and 10 (price-history retention): per-request CSRF tokens, TLS support, rate
limiting, backups, structured logs/monitoring, real email transport for password resets, and the
recorded (not executed) Postgres decision. Jay invoked `/build-phase W7` directly, which is itself the
signal the phase ladder's "Trigger: the first remote user / pilot deploy" column asks for — per
`.claude/rules/00-process.md`, building is never gated, and this phase was already on the approved,
recorded execution map (`website_production_overview.md` §6), not a speculative reach.

**What "hosting" means for a coding agent with no cloud credentials, no domain, and no ability to take
hard-to-reverse infrastructure actions autonomously.** I did not provision any real cloud resource,
register a domain, or execute an actual deploy — those are exactly the kind of hard-to-reverse,
credentialed, shared-system actions the platform's own operating discipline reserves for explicit human
authorization, and no specific hosting platform has been chosen yet (a real decision, not mine to make
unilaterally). What I built instead is the **code and configuration that makes a real deploy safe the
day one happens**: every hardening item that is actually testable and reviewable without live infra
(CSRF, rate limiting, TLS *support*, secure-cookie posture, structured logs, backup/retention
mechanics, real SMTP), plus a generic `Dockerfile` as the deploy-target artifact and a recorded (not
executed) hosting-platform posture. See "Explicitly Deferred" for the precise line between what's
built and what's recorded-only.

## Practices invoked, in all three domains

- **(a) Software/coding craft.** Centralizing security concerns in ASGI middleware (`CSRFMiddleware`,
  `RequestLoggingMiddleware`) rather than threading a `Depends()` through every route is the same
  "one place, not scattered" discipline `web/config.py`'s docstring states explicitly — it also means
  zero of the 46 pre-existing `client.post(...)` call sites needed editing, because the *bypass* is
  centralized too (one conftest fixture, not 46 edits). The double-submit-cookie CSRF design reuses W5's
  own precedent (`src/auth/credentials.py::generate_token`'s reasoning) that a random opaque token needs
  no signing secret to be unforgeable — no new secret to manage, mirroring "W5 already retired the one
  prior signing secret entirely."
- **(b) Data-science/statistical concept.** The retention window (400 days) is a deliberate
  bias-variance-style tradeoff against `src/pricing/trends.py`'s own lookback logic: too short and a
  year-over-year price comparison (the highest-value trend claim a chef would ask about) loses its
  denominator; too long and the seam leg grows forever for no analytical benefit, since neither
  `trends.py` nor (not yet) `forecasting/` needs data older than a season or two of comparison. The
  "always keep the latest observation per ingredient regardless of age" rule is the same "never silently
  drop the only estimate you have" instinct behind `unconstrain.py`'s censored-demand handling elsewhere
  in this codebase — a missing/stale price must degrade honestly, never vanish.
- **(c) Restaurant/consulting standard.** Rate-limiting the login and upload funnels while leaving
  `/confirm`/`/invoice/confirm`/`/menu-prices` unlimited mirrors how a real host stand doesn't slow down
  someone mid-task at the door — throttle the entry points a stranger could hammer, never the flow a
  known, already-authenticated chef is actively completing.

## Key Design Decisions

### Decision: stateless double-submit-cookie CSRF, not a session-scoped DB-stored token
- **Chose:** a random per-visitor `csrf_token` httponly cookie (`web/csrf.py`), required to match a
  submitted form field on every unsafe-method request, enforced by one `BaseHTTPMiddleware`.
- **Over:** adding a `csrf_token` column to the `sessions` table (session-scoped, server-verified
  against a stored value) — the more "textbook" synchronizer-token pattern.
- **Because:** the double-submit pattern needs no new migration, works identically pre-auth (`POST
  /login`) and post-auth (every other protected route) with one code path instead of two, and needs no
  signing secret — consistent with W5's own decision to retire the one prior signing secret entirely
  rather than persist a new one. A session-scoped token would also have left `POST /login` itself
  unprotected (no session exists yet at that point) without a separate pre-auth mechanism anyway.
- **Risk if this was wrong:** low — double-submit is a standard, well-vetted pattern; its one weakness
  (a same-site XSS that can read `document.cookie`) doesn't apply here because the cookie is
  `httponly`, so no XSS payload running in the browser can read it either, matching how a synchronizer
  token would fare under the same threat.

### Decision: CSRF enforcement applies to every state-changing route uniformly, no exemptions
- **Chose:** `CSRFMiddleware` checks every `POST`/`PUT`/`PATCH`/`DELETE`, including
  `POST /reset-password/{token}` (whose action is actually authorized by the URL-embedded bearer token,
  not by ambient session-cookie authority — the thing CSRF normally targets).
- **Over:** exempting that one route with a documented rationale.
- **Because:** an exemption is a second thing a future maintainer has to remember and re-justify every
  time this file changes; applying the check everywhere costs nothing (the route already renders through
  the same base template that gets the hidden field for free via `_nav_context`) and removes an
  interpretation question a reviewer would otherwise have to adjudicate.
- **Risk if this was wrong:** none — a bearer-token-authorized route gains a redundant check, not a
  broken one.

### Decision: existing tests bypass CSRF/rate-limiting by default via one conftest.py fixture, not per-call-site edits
- **Chose:** two autouse fixtures in `conftest.py` (`_bypass_csrf_by_default`, `_reset_rate_limits`)
  that patch the module-level check functions the middleware calls by bare name at request time — the
  same "read it fresh through the module" pattern `src/db/engine.py::get_db` already documents for
  `SessionLocal`. A new `test_web_csrf.py` overrides the bypass fixture (same name, module-local) to
  test the real enforcement.
- **Over:** adding a `csrf_token` field to all 46 pre-existing `client.post(...)` call sites across five
  test files.
- **Because:** those 46 call sites test upload/auth/menu-price/insights *logic*, not CSRF — the exact
  same reasoning `test_web_upload.py`'s pre-existing `_bypass_login` fixture already applies to the
  *auth* concern for that file. Touching 46 unrelated call sites for a cross-cutting concern this
  codebase already has a precedent for isolating would have been unnecessary churn (Anti-Drift: "build
  only this phase").
- **Risk if this was wrong:** low — the *real* enforcement is fully covered by `test_web_csrf.py`'s
  dedicated end-to-end tests (cookie issuance, missing/mismatched/matching token, a live multipart
  upload through the real middleware); the bypass only removes CSRF as a *variable* from tests that
  were never exercising it as their subject.

### Decision: rate-limit `/login`, `/reset-password`, `/upload`, `/invoice/upload` — not `/confirm`, `/invoice/confirm`, `/menu-prices`
- **Chose:** four buckets, budgeted generously (10/5/20/20 per 60-second window).
- **Over:** rate-limiting every mutating route uniformly.
- **Because:** the production overview names "rate limiting on the funnels" specifically; `/login` is
  the textbook brute-force target and `/reset-password` the textbook enumeration/spam target, so both
  earned a budget even though the overview's phrasing is funnel-specific. `/confirm`/`/invoice/confirm`
  continue a flow the operator already started at `/upload`/`/invoice/upload` (which is already
  budgeted) and `/menu-prices` is a routine editing action for an already-authenticated chef — budgeting
  those too would risk throttling a real workflow for no attacker-facing benefit.
- **Risk if this was wrong:** low — the buckets are independent and cheap to add to more routes later if
  abuse is observed there.

### Decision: retention window is 400 days, chosen and pruning is a periodic script, never inline on the write path
- **Chose:** `DEFAULT_RETENTION_DAYS = 400`, applied by `scripts/apply_retention.py`, never automatically
  on `/invoice/confirm`.
- **Over:** a shorter/arbitrary window, or pruning synchronously every time a new invoice is confirmed.
- **Because:** 400 days covers a full year-over-year comparison with headroom (see Practices (b) above);
  pruning inline would add latency to the one action a chef is actively waiting on and would make an
  idempotent, order-independent maintenance operation part of the request hot path for no reason.
- **Risk if this was wrong:** low — the window is a single named constant, trivially adjustable; the
  policy always keeps the latest observation per ingredient regardless of age, so even an overly
  aggressive window can't silently zero out the grid's current prices.

### Decision: real SMTP transport is added, but Postgres is not — both were "W7 decision points," treated differently
- **Chose:** build and wire real SMTP (`src/email/sender.py`) with a dev/test no-op fallback; leave the
  Postgres swap unexecuted, `ONRAMP_DATABASE_URL` unchanged from SQLite.
- **Over:** treating both identically (either build both or defer both).
- **Because:** the two "W7 decision points" have different actual triggers.
  `website_production_overview.md` §3 records Postgres as "execute the swap **iff** hosted concurrency
  demands it" — no hosted concurrency exists yet (no deploy), so executing it now would be unrequested,
  irreversible-feeling scope (a real schema/data migration for a need that doesn't exist). Email
  transport has no such conditional — the production overview's row 4 already knows it's needed the
  moment W7 lands, and W5's own review flagged the raw-token-in-logs behavior as a real risk
  (`docs/phase_decisions/W5_review.md` LOW-2) that this phase is explicitly scheduled to close. Building
  it thin (stdlib `smtplib`, no new dependency) and env-gated (no-op when unconfigured, so every existing
  test and today's localhost dev flow is unaffected) costs little and closes a named, real finding.
- **Risk if this was wrong:** none for Postgres (nothing changed, decision stays open and recorded, exactly
  as scheduled). For email: an SMTP failure with a host configured raises rather than silently
  reverting to the log-the-token fallback (see `src/email/sender.py`'s docstring) — a misconfigured
  production deploy fails loudly instead of quietly resurrecting the exact risk this phase closes.

### Decision: TLS support is env-driven cert/key *or* a documented reverse-proxy posture — no cloud platform chosen
- **Chose:** `web/config.py::resolve_tls_files`/`ensure_safe_bind` refuse a non-loopback bind unless
  `ONRAMP_TLS_CERTFILE`/`ONRAMP_TLS_KEYFILE` are both set; the recommended default path (documented, not
  enforced beyond the loopback-only fallback) is a reverse proxy / hosting-platform edge terminating
  TLS in front of a loopback bind.
- **Over:** committing to a specific hosting platform (Fly.io, Render, a bare VM + Caddy, ...) and
  writing its platform-specific config.
- **Because:** no pilot deploy or platform choice exists yet — writing platform-specific config now would
  be inventing a decision that isn't mine to make unilaterally, and the guard that matters structurally
  (never silently bind past loopback with zero TLS material) doesn't require knowing the platform.
  `Dockerfile` is the generic artifact that works under *either* future choice.
- **Risk if this was wrong:** none new — today's actual runtime behavior (bind `127.0.0.1`, plain HTTP)
  is completely unchanged from before this phase; the guard only fires if `ONRAMP_HOST` is ever set to
  something non-loopback without TLS material, which nothing in this repo does today.

## Constraints Encountered Mid-Build

- **`BaseHTTPMiddleware` + `request.form()` silently empties the downstream request body.**
  Starlette's `Request.form()` drains `request.stream()` without ever populating
  `Request._body`; `BaseHTTPMiddleware`'s `_CachedRequest.wrapped_receive` only replays a
  cached body to the app *if* `._body` was set (via `.body()`) — otherwise, once the stream is
  fully consumed, it replays an **empty** body "so downstream apps don't hang." Every protected
  POST 422'd with "field required" until `web/csrf.py::verify_csrf_request` was changed to call
  `await request.body()` before `await request.form()`. Verified via Starlette 1.3.1 source
  reading plus a live reproduction (both a urlencoded `/login` POST and a multipart `/upload`
  POST) before and after the fix.
- **Middleware registration order is reversed from call order.** Starlette's `add_middleware`
  inserts each new middleware at the *front* of the internal list, so the *last*-registered
  middleware ends up *outermost*. `CSRFMiddleware` is registered before
  `RequestLoggingMiddleware` so the latter wraps the former and logs every request — including
  the ones CSRF rejects with a 403 — confirmed by reading Starlette's
  `Starlette.build_middleware_stack` source directly rather than assuming registration order.
- **`uvicorn.run()`'s own logging setup does not clobber `configure_logging()`.** Confirmed by
  direct experiment (calling `uvicorn.Config(...).configure_logging()` after this phase's own
  `configure_logging()` in an isolated script, then logging again) that uvicorn's default
  `log_config` dict has no `"root"` key, so `logging.config.dictConfig`'s
  `disable_existing_loggers=False` behavior leaves the root logger (and this phase's
  `web.request` logger, which propagates to it) untouched. The structured JSON request log was
  proven live end-to-end via a real programmatic `uvicorn.Server` + an actual HTTP round trip
  (not just unit tests) after an unrelated shell/background-process output-capture artifact in
  this sandboxed dev environment made two earlier `python -m web` smoke-test attempts *appear*
  to produce no log output — isolating the cause conclusively ruled out a code defect.

## Reviewer Focus Areas

1. **The CSRF `request.body()`-before-`request.form()` fix.** This is the least obvious piece of
   code in the phase — it looks unnecessary (why read the body and discard it?) unless you know
   the `BaseHTTPMiddleware`/`_CachedRequest` replay mechanics. Worth confirming the reasoning in
   `web/csrf.py::verify_csrf_request`'s docstring is actually correct and not just
   coincidentally-working; I verified it two ways (Starlette source reading + a live
   before/after reproduction) but a second read is warranted given how easy this class of bug is
   to reintroduce (e.g. a future refactor that "simplifies" `verify_csrf_request` by removing
   the seemingly-redundant `.body()` call would silently reopen the 422 regression).
2. **The rate limiter's process-local state and the CSRF-bypass conftest fixture's blast
   radius.** `web/rate_limit.py`'s in-memory dict and `_bypass_csrf_by_default`'s autouse scope
   (repo-wide, not just `onramp/`) are both deliberate, documented simplifications — worth
   confirming neither is masking something the phase should actually have caught (e.g., whether
   the autouse bypass is scoped tightly enough that a real CSRF regression elsewhere in
   `web/app.py` couldn't hide behind it; I believe `test_web_csrf.py`'s dedicated coverage closes
   this, but it's the one place a scope decision and a test-coverage claim both rest on the same
   assumption).
