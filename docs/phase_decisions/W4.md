## Why this, why now

W0–W3 built the funnel and the two value surfaces (grid, insights) but left `/your-data` — the
on-ramp's one trust surface — at W2's deliberately minimal "counts only" state, and never revisited
it when W3 added a third seam leg (invoice/price history). `website_vision.md` §8 names W4 as exactly
this: deepen the transparency view into its full form and add the forecasting bridge panel, before
W5 stands up real identity/an app DB on top of it. Building the trust story now, on the existing
single-tenant PoC, means W5's real-user-facing surface inherits an honest page instead of launching
a from-scratch trust surface at the same time as new auth machinery.

## Practices invoked, in all three domains

- **Software/coding craft:** thin controller / glue-module separation held exactly to the existing
  pattern (`web/your_data.py` owns the derived counts, `app.py` stays a ~3-line route); the store
  helper's existing `FileNotFoundError`-as-normal-state contract was reused rather than reinvented for
  the third leg.
- **Data-science/statistical concept:** none new — this phase is presentation over already-computed
  aggregates (row counts, `nunique()`), not a new transform. The one discipline that *does* apply is
  epistemic honesty about model validity: the bridge panel describes what P0–P4 actually do (a
  calibrated quantile-to-prep-quantity turn) without citing the simulation-only dollar figures as if
  they were this operator's numbers — the same "don't confuse validated-on-synthetic-data with
  validated-on-real-data" discipline the engine's own docs enforce on themselves.
- **Restaurant/consulting standard:** transparency-as-sales-tool — showing a chef exactly what's held
  and why, in their language, is itself part of earning the one-time recipe sitdown (the on-ramp
  contract's "instant, dollar-legible trust" analog for a surface that has no dollar figure of its
  own). The firewall explanation is deliberately not a legal disclaimer voice; it's the same "yes,
  that's about right" plain-language standard rule 06 already applies to dollar figures, applied here
  to a trust claim instead.

## Key Design Decisions

### Decision: one page, not two
- **Chose:** extend the existing `/your-data` page with the firewall story, the three-leg ledger, and
  the bridge panel, rather than adding a second route (e.g. `/whats-next`).
- **Over:** a dedicated "What's next" page, matching `website_vision.md` §3's site-map listing "Your
  data" and "What's next" as two separate bullets under "D. Trust & handoff."
- **Because:** both are static, read-mostly trust content with no independent navigation need yet;
  adding a second nav item and route for a single evergreen panel felt like premature surface area
  this early, and the existing page already stacks multiple `<section>` blocks successfully (rule 05:
  don't over-build ahead of a concrete need for the split). Easy to split into two routes later — this
  is presentation structure, not stored data, so it costs nothing to defer.
- **Risk if this was wrong:** if a future phase wants the bridge panel to have its own nav-visible
  identity (e.g. marketing wants to link to it directly, or W8's storefront design pass wants "what's
  next" as a standalone landing anchor), splitting it out is a template + one route addition, not a
  data-model change — low-cost to reverse.

### Decision: price-leg stats read independently of BOM/sales existing
- **Chose:** `_price_leg_stats()` is called unconditionally in `build_your_data_summary()`, before the
  BOM/sales try/except, so `has_price_data` is correct even in the `has_data=False` (nothing captured)
  branch.
- **Over:** only checking the price leg inside the `has_data=True` path (mirroring how the original W2
  code was structured around a single try/except).
- **Because:** the invoice funnel (`/invoice/upload`) has no dependency on the capture funnel having
  run first — nothing in the route enforces that ordering. Assuming BOM/sales always precede invoices
  would have been an unverified ordering assumption exactly of the kind this codebase's discipline
  flags (see the CLAUDE.md standing orders on not inventing invariants the code doesn't enforce).
- **Risk if this was wrong:** none observed — the extra read is cheap (one Parquet file, same
  `FileNotFoundError`-as-normal-state contract already used everywhere else in this module) and the
  empty-state template branch doesn't even render the price fields, so this only matters if a future
  page reuses `has_price_data` outside the current gate.

### Decision: firewall explanation is unconditional (shown even with zero data captured)
- **Chose:** the "What we never touch" section renders regardless of `summary.has_data`.
- **Over:** gating it inside the same `{% if summary.has_data %}` block as the rest of the transparency
  content.
- **Because:** it's static prose with no data dependency, and rule 06 explicitly frames transparency as
  "a first-class surface, not fine print" — showing the trust story to an operator who hasn't captured
  anything yet costs nothing and plausibly helps conversion (seeing the firewall story *before*
  uploading is when it's most reassuring, not after).
- **Risk if this was wrong:** none functional; a reviewer could reasonably argue this belongs only in
  the "connected" state for a tighter, more minimal empty screen — a one-line template change either way.

### Decision: bridge panel makes no dollar claim and no timeline
- **Chose:** the "What this unlocks next" copy describes the engine's mechanism ("weighing the real
  cost of over-prepping against running out") without citing any of the simulation dollar figures
  (`forecasting/CLAUDE.md`'s ~$610/~11.7%/etc.) and explicitly states the engine "hasn't run on your
  restaurant's numbers yet."
- **Over:** citing a concrete percentage or dollar-savings estimate to make the pitch more compelling.
- **Because:** every one of those figures is validated only against a simulated restaurant, and both
  `CLAUDE.md` and `forecasting/CLAUDE.md` carry the explicit standing caveat that simulation numbers
  are "plausible placeholders, not validated facts" pending real customer discovery. Putting one in
  front of an actual (even if hypothetical-future) operator would be exactly the "confidently-wrong
  figure loses the chef on day one" failure rule 06 exists to prevent — just applied to a forward-
  looking claim instead of a backward-looking cost.
- **Risk if this was wrong:** none — this is the conservative direction; if anything, a reviewer may
  find the copy under-selling. That's the correct side to err on here.

## Constraints Encountered Mid-Build

- The structural boundary check (`tests/test_module_boundaries.py` /
  `test_seam_boundary_local.py`) greps for the literal substring `_truth` anywhere under `onramp/`,
  including comments and docstrings — not just code paths. My first draft used "raw/_truth firewall"
  in a docstring and "raw/_truth wall" in a test docstring to describe the concept in shorthand; both
  tripped the check and had to be reworded to "hidden-oracle firewall" / "hidden-oracle wall." This is
  a real, useful catch (rule 05's intent is that `onramp/` never even *names* the truth path, since a
  comment today is a stray import tomorrow) — flagging here so the reviewer knows it was hit and fixed
  during the build, not missed.
- `test_export_unknown_leg_returns_404` previously used `/your-data/export/prices` as its "this leg
  doesn't exist" example — adding the real `prices` export leg made that test assert the wrong thing
  (it would have started passing for the wrong reason: a 404 becoming a 200 silently flips a test's
  meaning if the assertion were on status code alone without inspecting *why*). Repointed it to
  `/your-data/export/nonexistent`.
- A `"1 observation across 1 ingredient"` string assertion in a new test broke because Jinja renders
  the count and the word "observation" on separate template lines (whitespace, not logic, whitespace
  collapses differently than in-editor line wrapping suggests) — loosened to two independent
  substring checks (matches the style already used elsewhen in this test file, e.g. checking `"120"`
  in `resp.text` rather than a whole phrase).

## Reviewer Focus Areas

1. **The one-page-not-two-pages call.** I bundled the transparency ledger and the bridge panel into
   the same `/your-data` view rather than giving the bridge panel its own route, reading the W4 phase
   row's "the full 'your data' transparency view **and** the forecasting 'what's next' panel" as one
   phase delivering two *sections*, not necessarily two *pages*. If the reviewer reads the site map
   (`website_vision.md` §3) as requiring two distinct navigable surfaces, this is the first place to
   push back.
2. **Honesty calibration on the bridge panel copy.** I deliberately kept the "what this unlocks next"
   language mechanism-only, no numbers, no timeline. It's possible this reads as *too* vague to be a
   compelling "on-ramp" pitch (on-ramp contract #1 is "instant, dollar-legible value" — this panel by
   design carries no dollar figure at all, since none can be honestly attached to this operator yet).
   Worth the reviewer's judgment on whether the copy still earns its place on a trust page versus
   reading as filler.
