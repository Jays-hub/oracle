## Key Design Decisions

### Decision: sync `def` route handler, not `async def`
- **Chose:** `@app.get("/") def grid(...)` (sync).
- **Over:** `async def grid(...)`.
- **Because:** `build_grid_data()` does blocking file I/O (`open`/`csv`/pandas via `src/`). A sync
  handler lets FastAPI/Starlette run it in a threadpool; an `async def` would run that blocking work
  directly on the event loop and stall every other request. Documented inline (`web/app.py:31-33`).
- **Risk if this was wrong:** at W0's traffic (single-restaurant demo) negligible either way; matters
  once W1+ adds concurrent tenant requests.

### Decision: margin displayed derives from the *rounded* cost, not the raw cost
- **Chose:** `margin_display = menu_price - round_to_quarter(raw_cost)` (`web/compute.py:107-110`).
- **Over:** displaying `margin = menu_price - raw_cost` next to a separately-rounded cost figure.
- **Because:** rule 06's reconciliation discipline — "Menu − ~Cost = Margin" must add up when a chef
  does the subtraction by eye. The code comment cites this as a **repaired regression** ("the very
  regression audit fix #5 repaired"), and `test_margin_reconciles_with_rounded_cost` guards it
  structurally, not just by convention.
- **Risk if this was wrong:** a credibility leak — displayed numbers that don't add up on day one, per
  `06-frontend-ux.md`'s explicit warning.

### Decision: uncostable dishes are surfaced in a named `skipped` list, never dropped
- **Chose:** collect `(name, reason)` into `skipped: list[SkippedDish]`, rendered in a `role="alert"`
  banner (`grid.html:10-18`).
- **Over:** `except ValueError: pass` (silently omitting the dish from the grid).
- **Because:** mirrors `src/run.py`'s CLI skip-collection so the web and terminal paths fail the same
  honest way (rule 01 missingness report, rule 07 name-the-failure). `test_uncostable_dish_is_surfaced_not_dropped`
  guards this as a named regression test, implying it was previously broken and fixed once already.
- **Risk if this was wrong:** a dish silently vanishing from a client-facing grid — worse than a
  visible error, since nothing signals data is missing.

### Decision: W0 reads local sample CSVs directly, not through `src/store.py` or the seam
- **Chose:** `web/compute.py` calls `src/bom/loader.py` and `src/pricing/compute.py` functions
  directly against `onramp/plate_cost/data/sample_*.csv`.
- **Over:** reading back through the seam (`data/raw/`) the way the engine would.
- **Because:** the seam is write-only from this module's perspective at this phase — `data/raw/`
  carries only the BOM + sales legs the engine needs, not menu prices or unit prices, so margins
  literally cannot be reconstructed from it. Reading the seam back is explicitly W2's job once there
  is captured tenant data (`web/compute.py:52-56`, `website_vision.md` §8).
- **Risk if this was wrong:** none currently — this is spec-mandated, not a judgment call.

## Open Roadmap Question Carried Forward (not a W0 edit)

`W0_review.md`'s lowest-priority finding (a NIT) flagged a pre-existing Phase-0 property, not a W0
regression: `src/report/grid.py`'s quadrant classification (`margin_high = r.margin >= mean_margin`)
uses **absolute dollar margin**, not the food-cost **ratio**. A high-ticket, thin-margin dish (e.g.
Ribeye Steak: $58 menu, 47% food cost — the "thin" badge) can still land in a high-margin quadrant
("Puzzle") because its dollar margin beats the mean, producing a card that reads "high-margin
quadrant" next to a "thin" badge — a mixed signal a chef may find contradictory. This was invisible
as a CLI-only tool; W0 making the grid client-facing is what surfaces it as worth a conscious call.

**Recorded, not fixed:** out of W0's scope (the property lives in Phase-0's `src/report/grid.py`, not
the web layer). Whoever next touches the grid's axis definition (or the site map's "Grid" /
"Dashboard" surfaces, `website_vision.md` §3) should decide — and record the decision — whether the
popularity × margin axis should switch to margin % (chef-legible ratio) or stay dollar-absolute, and/or
whether the page should surface the axis definition inline so the mixed signal above is explained
rather than silently carried forward.
